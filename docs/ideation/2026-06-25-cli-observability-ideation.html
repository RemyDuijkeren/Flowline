CLI Observability & Bug Reproduction

Focus: Flowline CLI runs on local developer machines and build servers. When bugs occur, there is no persistent diagnostic trace, making reproduction dependent on users re-running with --verbose. How can we capture as much information as possible — automatically, regardless of network availability?

Grounding Context

Codebase — What Exists

  • ExitCode enum (13 codes, stable public API)
  • FlowlineException.WithDetail(Action<IAnsiConsole>) hook → global handler renders it
  • --verbose flag → Console.Verbose() dimmed output; RuntimeOptions.IsVerbose propagated to all commands
  • CliWrap stdout/stderr piped per-line via PipeTarget.ToDelegate in WithToolExecutionLog
  • ToolCheckResult.Version cached from EnsureDotNet/EnsurePacCli/EnsureGit (never surfaced to user)
  • %LOCALAPPDATA%/Flowline/ already in use (validation-cache.json) with XDG_CACHE_HOME cross-platform fallback
  • RedactSensitiveArgs regex handles --client-secret and /mfaClientSecret:

Confirmed Gaps

  • No persistent log files — zero trace after terminal closes
  • No ILogger, no structured logging, no telemetry
  • Subprocess stderr discarded in non-verbose runs; always requires re-run with --verbose
  • No correlation IDs — multi-step CI pipelines produce disconnected log lines
  • No environment state snapshots — tool versions never surfaced on failure
  • FlowlineException.WithDetail used ad-hoc (not systematic) — some exit paths log nothing
  • Spectre spinners emit ANSI escape sequences that CI log renderers show as garbage

External Prior Art

  • dotnet CLI: opt-out telemetry, CI env-var detection list, mandatory TelemetryClient.FlushAsync() at exit
  • npm: rotating log files in ~/.npm/_logs/, prints log path to stderr on failure automatically
  • cargo: JSONL timing logs to ~/.cargo/log/, one file per invocation with unique ID
  • Tailscale / velero: --bug-report support bundle (Collectors → Redactors → archive)
  • DO_NOT_TRACK=1 convention: respected by GitHub CLI, npm

Technology Approach — Why Not ILogger?

  • None of I1–I5 use ILogger / Microsoft.Extensions.Logging. Flowline has none today and these ideas don't add it.
  • I1 (JSONL record) is an audit pattern, not a log stream. One structured JSON record per command invocation — command, args, exit code, duration, tool versions. A direct File.AppendAllTextAsync + JsonSerializer call is all it needs. Adding ILogger, a provider registration, and DI wiring for a single append per run is over-engineering.
  • ILogger fits a different (heavier) use case: verbose debug tracing where every internal step emits a line. That pattern suits long-running services. If Flowline ever needs step-level debug output, ILogger with a rolling file sink (Serilog, Microsoft.Extensions.Logging.File) would be the right tool — but that is a separate feature not in scope here.
  • I6 (App Insights): direct TelemetryClient is simpler than the Microsoft.Extensions.Logging.ApplicationInsights ILogger provider for a CLI. Both require mandatory FlushAsync() at process exit; direct client keeps the flush call explicit and visible.

CI / Build Server Constraints

  • Enterprise GitHub Actions: L7 egress firewall may block dc.services.visualstudio.com
  • No interactive TTY — first-run disclosures must write to stderr and not block
  • Ephemeral disk on GitHub-hosted runners — local files fine for job duration, not across runs
  • Secrets in CI logs: args may contain connection strings, PAC auth tokens — must redact
  • App Insights: InMemoryChannel + mandatory FlushAsync() — buffer fires every 30s or 500 events, neither fires in sub-second CLI runs

I1.

Always-On Structured Run Log

Local Persistence Confidence 90% Complexity S/M

Every command invocation appends a single JSONL record to %LOCALAPPDATA%/Flowline/runs/<date>.jsonl — written always, regardless of success or failure, without any flag. Each record contains: UTC timestamp, command name, args (redacted via existing RedactSensitiveArgs), exit code, total duration, Flowline version, cached tool versions (dotnet, pac, git), and — on failure — the full exception chain. Log rotates by calendar day, 30-day retention, oldest deleted on startup. On failure, the global exception handler prints one dim line: "Run log: %LOCALAPPDATA%\Flowline\runs\<today>.jsonl". The write is a fire-and-forget side-effect in Program.cs — zero latency impact on normal execution.

Every invocation produces a JSONL record. On failure, the path is printed. No flags, no re-runs.

Basis direct: ValidationCacheStore.cs already writes to %LOCALAPPDATA%/Flowline/ with cross-platform fallback chain (XDG_CACHE_HOME → ~/.cache → LOCALAPPDATA → TempPath) — the directory is confirmed writable. Program.cs global exception handler is the single capture point for all unhandled FlowlineExceptions — attaching a file write there requires no changes to any command. CommandExtensions.cs WithToolExecutionLog already accumulates stdout/stderr per-line via PipeTarget.ToDelegate — buffering into the record on failure is additive. external: npm rotating log pattern (~/.npm/_logs/, prints path on failure automatically).
Rationale This is the foundation all other ideas build on. A future --bug-report zip reads from it. Correlation IDs stamp it. CI detection writes to it unconditionally. Without a persistent log, every bug report starts at zero. With it, the previous run's full trace is available immediately after the first failure — no user action required.
Downsides Disk writes on every run (minimal: one JSON line). Log files accumulate on developer machines (~30-day rotation mitigates). Secrets must be carefully redacted — extend RedactSensitiveArgs to cover URL-embedded tokens and connection strings before logging any args. On air-gapped CI with no persistent disk across runs, the log is ephemeral (fine for the duration of a job).
I2.

Subprocess Stderr Capture Into Exception Detail

Diagnostic Capture Confidence 88% Complexity S

When a CliWrap subprocess (pac CLI, dotnet, git) exits with a non-zero code, the subprocess's output is currently lost unless --verbose was set before the failure. Add a rolling 50-line buffer inside WithToolExecutionLog that accumulates stderr (and last N stdout lines) regardless of verbose mode. When CliWrap throws on non-zero exit, attach that buffer as the FlowlineException.WithDetail callback. The user sees the actual subprocess failure reason immediately — on screen and in the JSONL log — without needing to re-run with any flag. The global handler already renders WithDetail; no new wiring required.

Basis direct: CommandExtensions.cs WithToolExecutionLog: non-verbose path writes stderr output to the console display but does not buffer it — the data is available per-line via PipeTarget.ToDelegate but discarded after display. Adding a List<string> accumulator (capped at 50 lines) to the existing pipe delegates is additive. FlowlineException.WithDetail(Action<IAnsiConsole>) — the hook is confirmed in Program.cs and renders on failure. SyncCommand.cs — on !result.IsSuccess, the current exception message explicitly defers to --verbose rather than including failure context. reasoned: The most common support ask ("re-run with --verbose and paste output") disappears when the buffer is always populated.
Rationale This is the highest-leverage single change for bug reproduction. PAC CLI auth failures, dotnet build errors, and git failures all produce subprocess output that currently requires a re-run to capture. One change to WithToolExecutionLog propagates to every caller simultaneously — Clone, Sync, Deploy, Push, Generate — because they all go through the same helper. On CI where the spinner was already suppressed, this is the only mechanism that surfaces subprocess failure reasons. Extension: when ConsoleHelper.IsInteractive() returns false (CI env already detected via CI, TF_BUILD, JENKINS_URL), stream all subprocess stdout/stderr unconditionally in real time rather than only buffering on failure — pipeline operators need the full trace regardless of exit code, not just the last 50 lines attached to an exception.
Downsides 50-line buffer holds data in memory for the duration of each subprocess call — minimal cost. The buffer must be flushed to the JSONL log (I1) on failure, not just printed, otherwise CI jobs that lose the terminal get no trace. Subprocess output can contain secrets (connection strings in pac CLI args) — apply RedactSensitiveArgs before writing to log. Note: the verifier confirmed stderr is currently printed to the display in non-verbose runs, but not persisted — the goal of this idea is persistence and injection into the exception's Detail callback, not suppressing display.
I3.

ILogger Debug File Sink

Diagnostic Capture Confidence 85% Complexity M

Register Microsoft.Extensions.Logging with a rolling file sink (Serilog or Microsoft.Extensions.Logging.File) writing to %LOCALAPPDATA%/Flowline/debug/<date>.log. Wire ILogger<T> into Flowline.Core services that process complex data: PluginService, WebResourceService, SolutionDiffService, and any service that iterates components or compares state. Log level: Warning by default; Debug when --verbose is active — RuntimeOptions.IsVerbose already propagates, wiring it to the minimum log level is one line. The debug file fills the gap I1 and I4 can't cover: what specific data was being processed at the moment of failure — which plugin step, which solution component, which comparison failed. I5's crash bundle picks up the debug log file automatically.

Basis direct: Flowline has no ILogger today (confirmed gap). Flowline.Core confirmed to have complex in-process logic — PluginService (step registration comparison), WebResourceService (file sync), SolutionDiffService — with no diagnostic trace beyond exception stack frames. RuntimeOptions.IsVerbose already propagated via FlowlineCommand.InitializeRuntimeOptions — adding a log-level switch on it is one conditional in the logging config. %LOCALAPPDATA%/Flowline/ confirmed writable; the debug subdirectory uses the same path resolution as I1. I5 (crash bundle) zips the runs directory — extend it to include the debug log file. external: Microsoft.Extensions.Logging is the .NET standard logging abstraction; Serilog rolling file and Microsoft.Extensions.Logging.File are established providers. dotnet CLI uses MEL throughout its command implementations for exactly this purpose. reasoned: ILogger<T> constructor injection is the standard .NET pattern — once the DI registration exists, each service adds one constructor parameter. Return per new log statement is high in data-processing loops where a single LogDebug("Comparing step {Step} on {Plugin}: registered={R}, current={C}", ...) call makes an otherwise opaque diff failure immediately reproducible.
Rationale I1 and I2 cover the two most common failure classes — subprocess failures and the JSONL audit record. But commands that run heavy Flowline.Core logic (Deploy, Clone, Provision) fail in a third way: in-process data-dependent logic where the exception message alone is not reproducible. A crash in PluginService comparing step registrations gives you a stack frame but not which plugin, which step, or what attribute mismatch caused it. LogDebug calls at data-boundary decision points — before/after comparisons, at loop entry with the item being processed — make these failures immediately reproducible without a second run. The --verbose connection keeps the default run quiet; the debug file only grows when explicitly requested.
Downsides Constructor injection changes across Flowline.Core services — mechanical but proportional to how many services are instrumented. Don't add LogDebug speculatively to every line — add it at data-boundary decision points: before/after comparisons, at loop entry with the item, at conditional branches with the condition values. Debug log files accumulate alongside I1's JSONL records — apply the same 30-day rotation. On CI with --verbose not set, the debug file stays empty; the runner's log is the stdout trace from I2.
I4.

DiagnosticContext Stage Chain

Diagnostic Capture Confidence 85% Complexity M

Commands like Deploy, Sync, and Clone execute multiple named stages: connect to Dataverse, validate solution, run PAC sync, pack solution, drift check. Today, when a failure occurs mid-command, the exception says "Deploy failed" but not which stage failed. Introduce a lightweight DiagnosticContext — a List<string> scoped to the command execution lifetime — that commands populate at each named stage: context.Stage("Connecting to Dataverse"), context.Stage("Running pac solution sync"). On FlowlineException throw, the global handler flushes the chain via a registered WithDetail callback, printing the stages completed and the one that failed. No-op on success. Single registration point in FlowlineCommand.ExecuteAsync.

Basis direct: FlowlineCommand.ExecuteAsync is the single entry point for all commands — confirmed. One place to inject and dispose the DiagnosticContext. FlowlineException.WithDetail is already wired to the global handler; no new rendering code needed. The gap is confirmed: Detail callback pattern is ad-hoc (Gap #8 in dossier) — some exit paths log nothing to Detail (CloneCommand exit codes, for example). reasoned: A List<string> scoped to the command avoids thread-safety concerns since commands execute sequentially. This is the lowest-complexity way to add structured stage tracking without introducing a logging framework.
Rationale Without a stage chain, a Deploy failure tells you the command failed but not whether it failed at connection, version bump, PAC sync, pack, or drift check — five different scenarios with five different root causes. The stage chain collapses these into one diagnostic line: "Completed: Connect, Validate, PackSolution. Failed at: PAC sync." This replaces the ad-hoc WithDetail pattern with a consistent mechanism every command automatically participates in once the base class is wired up. Stages also appear in the JSONL log record (I1), giving CI operators a machine-readable sequence to grep.
Downsides Requires instrumenting each command's stage boundaries — mechanical work but proportional to command count. Stage names are strings; a future enum or constant pool would be cleaner but is not required now. The DiagnosticContext must be disposed cleanly even when FlowlineException isn't thrown (normal-exit path), to avoid memory leaks on long-running sessions that run many commands.
I5.

Crash-Initiated Support Bundle

Bug Report Packaging Confidence 87% Complexity M

The assumption broken: the user initiates bug reporting. Today's global exception handler (default: branch in Program.cs for unhandled exceptions) prints a stack trace and exits — discarding the environment state at the moment of failure. Flip: on any unhandled exception, the handler atomically writes a zip to %LOCALAPPDATA%/Flowline/bug-reports/<timestamp>.zip containing: last N JSONL run records, the .flowline config (environment URLs and solution names — connection strings redacted), PAC auth profile names (not credentials), the env fingerprint (tool versions from the cached ToolCheckResult), and the full exception with stack trace. The handler prints a single line: "Bug report saved: <path> — attach to issue at github.com/...". Zero user effort.

Assumption flipped: the tool creates the bug report, not the user. Attaching the zip replaces pasting a stack trace.

Basis direct: Program.cs default: branch for unhandled exceptions is confirmed — currently only calls AnsiConsole.WriteException(ex, ExceptionFormats.ShortenPaths) then exits; this is exactly where the bundle write attaches. RedactSensitiveArgs in CommandExtensions.cs is confirmed and reusable. %LOCALAPPDATA%/Flowline/ is confirmed writable. external: dotnet CLI crash telemetry (automatic on crash, no user action). Velero debug bundle and Tailscale bugreport model (auto-package then hand to user).
Rationale Most unhandled Flowline exceptions come from edge-case PAC CLI state, MSAL cache mismatches, or Dataverse auth errors in environments where the developer didn't run --verbose. Every such failure currently produces zero useful diagnostic output beyond the exception message. The bundle-on-crash captures the exact environment state at the moment of failure — with no user action — and transforms the support conversation from "can you reproduce this?" to "attach the zip." The %LOCALAPPDATA%/Flowline/bug-reports/ directory accumulates bundles over time; old ones can be pruned on startup.
Downsides File I/O in an exception handler must be robust — wrap in try/catch so a write failure doesn't mask the original exception. Zip creation adds ~50-100ms to crash paths (acceptable). Disk accumulation: bundle files are large (~50KB each) and developers may not notice them; add auto-prune at startup (keep last 10) or at least document the path. Requires I1 (JSONL log) to exist for the run entries — ship I1 first.
I6.

Per-Invocation Correlation ID

Diagnostic Capture Confidence 82% Complexity S

On startup, read FLOWLINE_TRACE_ID from the environment (CI pipelines set this to their run ID: ${{ github.run_id }} in GitHub Actions, $(Build.BuildId) in Azure DevOps). If absent, auto-generate an 8-char hex ID. Store on FlowlineRuntimeOptions — same threading pattern as IsVerbose — and stamp every verbose output line and every JSONL log record with it. When two verbose runs of the same command produce different outcomes, diffing the two log records becomes a grep. When a CI pipeline runs flowline clone then flowline deploy, both records share the pipeline run ID and are trivially joinable.

Basis direct: RuntimeOptions.IsVerbose is confirmed as a bool propagated to all services via FlowlineCommand.InitializeRuntimeOptions — adding a string TraceId follows the identical threading pattern. The gap is confirmed: no correlation ID exists in the current codebase (Gap #1 in dossier). external: FLOWLINE_TRACE_ID follows the convention of W3C traceparent, OTEL correlation IDs, and dotnet CLI's CI detection via env vars — env-var correlation is the established pattern for developer CLIs. Git's GIT_TRACE per-invocation env var is the closest analogy. reasoned: FlowlineConsoleExtensions.Verbose() already has the right shape to accept an optional trace ID parameter (adding one with a null default is non-breaking).
Rationale CI pipelines often run multiple Flowline commands sequentially. Without a shared ID, the JSONL log from a GitHub Actions workflow contains N disconnected records — you can't tell which flowline deploy record belongs to which pipeline run. With FLOWLINE_TRACE_ID=${{ github.run_id }} in the workflow env block, every record from that pipeline run is instantly filterable: grep "$GITHUB_RUN_ID" ~/.local/share/Flowline/runs/2026-06-25.jsonl. Pairs with I1 to complete the CI observability story.
Downsides CI operators have to know to set FLOWLINE_TRACE_ID — this is opt-in (auto-generation handles the common case, but CI correlation requires the env var). Auto-generated IDs are random and not meaningful without a log; developers looking at a JSONL record see an opaque hex string. The feature's value is proportional to how consistently users look at log files — low if I1 isn't shipped first.
I7.

App Insights Opt-Out Telemetry

Remote Telemetry Confidence 65% Complexity L

On unhandled exceptions or crash exits: send a minimal telemetry event to Azure Application Insights — exception type name, exit code enum value, command name (no args), Flowline version, OS/arch, CI flag. No PII: no file paths, no user identifiers, no Dataverse URLs. Disclose at first run to stderr (CI mode suppresses disclosure and proceeds silently). Opt-out via FLOWLINE_TELEMETRY_OPTOUT=1 or DO_NOT_TRACK=1. Use InMemoryChannel (not ServerTelemetryChannel) and call TelemetryClient.FlushAsync() before process exit — this is mandatory for short-lived CLI processes since the 30s/500-event buffer never fires. Offline queue: on network failure, append to %LOCALAPPDATA%/Flowline/telemetry-queue.ndjson and retry on next invocation.

Basis external: dotnet CLI's opt-out model (DOTNET_CLI_TELEMETRY_OPTOUT, first-run disclosure to stderr, CI env-var detection) is the established precedent for .NET developer tools. InMemoryChannel + mandatory TelemetryClient.FlushAsync() at process exit is the documented pattern for short-lived CLIs — without FlushAsync, telemetry is lost. DO_NOT_TRACK=1 convention is respected by GitHub CLI and npm. ServerTelemetryChannel stores events to disk during network outages for retry. reasoned: Local logs (I1) surface bugs that users bother to report. Telemetry surfaces crash patterns from users who never file issues — especially CI failures on build servers that fail silently.
Rationale Local log files only tell you about bugs users report. Telemetry reveals what's actually breaking across all users — the silent majority of crashes that never turn into GitHub issues. For a developer tool in early adoption, aggregate crash data (which commands fail most, which exit codes are hit in the wild, whether CI environments behave differently from local) is strategically valuable for prioritization. The store-and-forward queue makes it resilient to the CI egress firewall constraint — events accumulate locally and flush when connectivity allows.
Downsides Enterprise egress constraint (major): enterprise GitHub Actions and Azure DevOps self-hosted agents may have L7 egress firewalls that block dc.services.visualstudio.com — the App Insights ingestion endpoint. This is a confirmed constraint in the grounding. Telemetry will silently fail for these users and must never block or slow the CLI. Consent controversy: GitHub CLI's April 2026 opt-out telemetry rollout drew significant community backlash — disclosure clarity and easy opt-out matter more than the mechanism. Complexity: requires Azure App Insights resource, connection string distribution, and ongoing maintenance. Recommend shipping I1–I6 first; telemetry is most valuable when local logging is already in place and you want aggregate signal beyond what local logs provide.
Sequencing Advice

Implementation Waves

Four independently shippable waves. Each builds on the last — Wave 1 alone is already meaningful value. The groupings follow code locality and dependency, not just priority.

Wave 1 — Foundation I1 + I2 + I3 Do together · single PR
I1 — Always-On JSONL Log Persistent per-invocation audit record. Every other idea writes to or reads from this file.
I2 — Subprocess Stderr Capture Rolling buffer flushes to I1's log on failure. Needs I1 to be more than a screen print.
I3 — ILogger Debug File Sink In-process detail for Flowline.Core commands. Same DI setup pass as I1; same output directory. Without this, heavy-logic commands (Deploy, Clone, Provision) still have blind spots.
Why together: I1 and I3 share the same DI registration pass and output directory. I2's buffer flush target is I1's log. Splitting them ships an incomplete foundation.
Wave 2 — Rich Context I4 + I6 Small PR · same injection point
I4 — DiagnosticContext Stage Chain Adds "which stage failed" to every exception. Registers in FlowlineCommand.ExecuteAsync.
I6 — Per-Invocation Correlation ID Stamps every JSONL record and verbose line with a trace ID. Adds one field to RuntimeOptions.
Why together: both enrich what the Wave 1 log captures. Both touch FlowlineCommand.ExecuteAsync and RuntimeOptions — natural single pass. I6 is near-trivial (one string field); don't ship it in isolation.
Wave 3 — Packaging I5 Focused PR · after Wave 1
I5 — Crash-Initiated Support Bundle On unhandled exception: zip I1's JSONL records + I3's debug log + redacted config → write to bug-reports/ → print path. One focused change to Program.cs default: branch. Depends on I1 existing; I3's debug log makes the bundle substantially more useful. Ship after Wave 1; Wave 2 can precede or follow.
Wave 4 — Telemetry I7 Separate product decision
I7 — App Insights Opt-Out Telemetry Lowest confidence (65%), highest complexity (L). Requires Azure resource + connection string distribution + ongoing maintenance. Its value — aggregate crash signal across all users — only materialises after Waves 1–3 are deployed and you want patterns that local logs can't show. Treat as a separate product decision, not part of the observability rollout.
