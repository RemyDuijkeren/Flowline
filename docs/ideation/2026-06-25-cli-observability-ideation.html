CLI Observability & Bug Reproduction

Focus: Flowline CLI runs on local developer machines and build servers. When bugs occur, there is no persistent diagnostic trace, making reproduction dependent on users re-running with --verbose. How can we capture as much information as possible — automatically, regardless of network availability?

Status (2026-06-29): Waves 1 and 1.5 shipped. ILogger/Serilog file sink, subprocess capture buffer, and console-to-ILogger tee all in production. Waves 2–4 planned below.

Phase 1 Evidence · Updated post-implementation

Grounding Context

Codebase — What Exists Now

  • ExitCode enum (13 codes, stable public API)
  • FlowlineException.WithData — replaced WithDetail; stores arbitrary key/value context surfaced by global handler
  • --verbose flag → Console.Verbose() dimmed output; RuntimeOptions.IsVerbose propagated to all commands
  • FlowlineRuntimeOptions.VerboseOutput — rolling 50-line buffer; fills on non-verbose runs; flushed to terminal on failure
  • CliWrap stdout/stderr piped per-line via PipeTarget.ToDelegate in WithToolExecutionLog (passes RuntimeOptions, fills VerboseOutput)
  • ILogger / MEL + Serilog file sink — one <timestamp>Z.log per invocation in %LOCALAPPDATA%/Flowline/logs/
  • ILogger<T> injected into PluginService, WebResourceService, SolutionDiffService; FlowlineCommand base has lazy Logger property
  • LoggingRenderHook (IRenderHook) — tees every AnsiConsole Markup write to ILogger; level detected from visual prefix
  • Visual prefixes: Info → · , Skip → ↷ , Ok → ✓ , Done → 🚀
  • FlowlineStoragePaths — resolves %LOCALAPPDATA%/Flowline/ with XDG_CACHE_HOME cross-platform fallback; used by ILogger and exception handler
  • RedactSensitiveArgs regex handles --client-secret and /mfaClientSecret:
  • ToolCheckResult.Version cached from EnsureDotNet/EnsurePacCli/EnsureGit

Remaining Gaps

  • No correlation IDs — multi-step CI pipelines produce disconnected log files
  • No stage chain — failures say "Deploy failed" not "failed at PAC sync stage"
  • No persistent support bundle — on crash, no zip is created automatically
  • No remote telemetry — crash patterns invisible across users
  • Some Wave 1 ILogger call sites now duplicate terminal output captured by the tee — cleanup deferred
  • Verbose messages suppressed when --verbose off — hook never sees them; crash context handled by VerboseOutput buffer only

What Was Originally a Gap (Now Closed)

  • No persistent log files → ILogger/Serilog file sink (one file per invocation)
  • No ILogger, no structured logging → MEL + Serilog + LoggingRenderHook
  • Subprocess stderr discarded on non-verbose failure → VerboseOutput rolling buffer, rendered on failure
  • Spectre spinners emit ANSI garbage in CI → IRenderHook filter excludes non-Markup renderables
  • Dual writes required (console + logger) → LoggingRenderHook observes terminal, no per-call opt-in

CI / Build Server Constraints

  • Enterprise GitHub Actions: L7 egress firewall may block dc.services.visualstudio.com
  • No interactive TTY — first-run disclosures must write to stderr and not block
  • Ephemeral disk on GitHub-hosted runners — local files fine for job duration, not across runs
  • Secrets in CI logs: args may contain connection strings, PAC auth tokens — must redact
  • App Insights via OTLP: TracerProvider.ForceFlush() + dispose before exit — replaces InMemoryChannel/TelemetryClient.FlushAsync() from the classic SDK; must complete before Environment.Exit

I1.

Always-On Structured Run Log

Local Persistence Dropped

Originally: append a single JSONL record per invocation to %LOCALAPPDATA%/Flowline/runs/<date>.jsonl — command name, args, exit code, duration, tool versions, exception chain on failure.

Why Dropped The ILogger/Serilog file sink (I3) already captures the full narrative, exception, and stack trace in one file per invocation. A parallel JSONL record would be a redundant second artifact — same data, two files. I3 subsumes I1's use case: "user sends logs for issue triage." One file is simpler than two. The JSONL format's machine-readability advantage disappears when the primary consumer is a human reading a bug report.
I2.

Subprocess Stderr Capture Into Failure Output

Diagnostic Capture Confidence 88% Complexity S Shipped · Wave 1

Rolling 50-line VerboseOutput buffer on FlowlineRuntimeOptions accumulates subprocess output (and Flowline's own verbose lines) regardless of verbose mode. On non-zero subprocess exit, the buffer is rendered to the terminal between the Flowline error message and the dim Log: path line. When --verbose was already active, the buffer is suppressed — output was already printed live. Buffer fills via the FlowlineConsoleExtensions.Verbose(console, message, options) overload and the updated WithToolExecutionLog(RuntimeOptions, ctx) overload.

Shipped as FlowlineRuntimeOptions.VerboseOutput (rolling queue, cap 50, Append/Lines/Clear). New Verbose(IAnsiConsole, string, FlowlineRuntimeOptions) overload: buffers when non-verbose, writes live when verbose. WithToolExecutionLog gains FlowlineRuntimeOptions overload; all command call sites migrated. Exception handler in Program.cs renders runtimeOptions.VerboseOutput.Lines in dim before the Log: line.
Differs from original Original idea attached the buffer to FlowlineException.WithDetail. Actual implementation keeps the buffer on RuntimeOptions and reads it directly in the exception handler — no changes to FlowlineException needed. Simpler: the exception handler already had the right access point.
I3.

ILogger Debug File Sink

Diagnostic Capture Confidence 85% Complexity M Shipped · Wave 1

MEL + Serilog file sink writing to %LOCALAPPDATA%/Flowline/logs/<yyyy-MM-ddTHHmmss>Z.log — one file per invocation, always-on, no flag required. Minimum level: Debug; Serilog overrides: Warning for Microsoft.* and System.* (suppresses Dataverse client noise). ILogger<T> injected into PluginService, WebResourceService, and new SolutionDiffService. FlowlineCommand base class exposes a lazy protected ILogger Logger property. On failure the global exception handler prints a dim Log: <path> line.

Shipped as Serilog packages: Serilog, Serilog.Extensions.Logging, Serilog.Sinks.File. Microsoft.Extensions.Logging.Abstractions in Flowline.Core.csproj. FlowlineStoragePaths helper for path resolution (mirrors ValidationCacheStore.GetDefaultCachePath()). SolutionDiffService created as a thin wrapper around SolutionChangeSummary.ComputeAsync. Log.CloseAndFlush() after app.RunAsync. Wave 1 adds ~5–8 LogInformation milestones across the three services.
Note Wave 1 milestone LogInformation calls now partially overlap with what the LoggingRenderHook (I3b) captures from terminal output. Follow-up: audit and remove redundant explicit logger calls from Wave 1 services — deferred to Wave 2.
I3b.

Console-to-ILogger Tee via LoggingRenderHook

Diagnostic Capture New idea Shipped · Wave 1.5

Not in the original ideation. Emerged during Wave 1 implementation: every console extension method (Ok, Info, Skip, Warning, Error) produced terminal output that still required a companion logger.Log(...) call to appear in the file. Rather than adding dual writes everywhere, attach a single IRenderHook to AnsiConsole.Console.Pipeline that observes all terminal Markup writes, renders to plain text via a no-color console, and logs at the detected level. Terminal output is unaffected — the hook yields each renderable through before logging.

One call writes to both terminal and log. Hook yields first (render pipeline unaffected), logs second. Non-Markup renderables (spinners, progress bars) pass through with zero logging.

Level detection After stripping Spectre markup: ✓ / · / ↷ / 🚀 → Information · Warning: → Warning · Error: → Error · empty/whitespace → skip · anything else → Debug. Info got new · prefix; Skip got ↷ prefix to make them recognisable.
Key constraints yield return before the try/catch — logging failure must never abort the render pipeline. Render at int.MaxValue width (not console width) — narrow CI terminals cause word-wrap, splitting one message across multiple log entries. ILoggerFactory created outside DI container (CommandApp doesn't expose IServiceProvider before RunAsync) and disposed after Log.CloseAndFlush().
Solution doc docs/solutions/architecture-patterns/spectre-console-ilogger-render-hook.md — full implementation with gotchas, test pattern, and when-to-apply guidance.
I4.

DiagnosticContext Stage Chain

Diagnostic Capture Dropped

Originally: a List<string> scoped to command execution, populated at each named stage, flushed to the failure output on exception. Dropped in favour of ActivitySource / Activity (absorbed into I6).

Why Dropped Two reasons. First, LoggingRenderHook (I3b) already tees every terminal write to the log file sequentially — if a command prints "Connecting to Dataverse…", "Running PAC sync…", the log already has that trail; a collapsed stage summary in the exception output adds marginal value. Second, DiagnosticContext is a Serilog ASP.NET Core concept (request-scoped enricher for web middleware pipelines), not a .NET standard, and a poor fit for a single-invocation CLI. The same stage-chain value — and more — comes for free from ActivitySource child activities (I6): each stage becomes a named span with duration, status, and W3C-standard trace correlation, without any custom type.
I5.

Crash-Initiated Support Bundle

Bug Report Packaging Confidence 87% Complexity M Planned · Wave 3

On any unhandled exception, the handler atomically writes a zip to %LOCALAPPDATA%/Flowline/bug-reports/<timestamp>.zip containing: last N ILogger log files, the .flowline config (environment URLs and solution names — connection strings redacted), PAC auth profile names (not credentials), cached tool versions, and the full exception with stack trace. The handler prints a single line: "Bug report saved: <path> — attach to issue at github.com/...". Zero user effort. Now supersedes the original JSONL bundle idea (I1 is gone) — the ILogger files become the primary log content in the zip.

Basis Program.cs default: branch for unhandled exceptions is confirmed — currently only calls AnsiConsole.WriteException then exits; this is exactly where the bundle write attaches. RedactSensitiveArgs in CommandExtensions.cs is confirmed and reusable. FlowlineStoragePaths already exists and resolves the correct base path. ILogger files (I3) are the log source — the bundle zips the most recent few from logs/.
Rationale Most unhandled Flowline exceptions come from edge-case PAC CLI state, MSAL cache mismatches, or Dataverse auth errors. Every such failure currently produces zero useful diagnostic output beyond the exception message. The bundle-on-crash captures the exact environment state at the moment of failure and transforms the support conversation from "can you reproduce this?" to "attach the zip." With I3 shipped, the ILogger file already captures the full terminal session — the bundle just packages it.
Downsides File I/O in an exception handler must be robust — wrap in try/catch so a write failure doesn't mask the original exception. Zip creation adds ~50-100ms to crash paths (acceptable). Disk accumulation: prune to last 10 bundles on startup.
I6.

Per-Invocation Trace via ActivitySource

Diagnostic Capture Confidence 88% Complexity S Planned · Wave 2

In FlowlineCommand.ExecuteAsync, start a root Activity from a shared FlowlineActivitySource named after the command. This gives a W3C-standard 128-bit trace ID (Activity.Current?.TraceId) propagated automatically through all async code via AsyncLocal — no manual threading. Each named stage (connect, PAC sync, pack, drift check) starts a child Activity, absorbing the stage-chain value of the dropped I4 with a richer result: named spans with duration and status, not a flat string list. The root activity carries command-level tags: flowline.version, ci, ci.platform, os.arch — these flow to App Insights (I7) as searchable custom dimensions for cross-user and build-server analysis.

CI pipelines chain runs by setting a standard W3C TRACEPARENT env var (e.g. TRACEPARENT=00-${{ github.run_id }}-...); Activity picks this up automatically. No custom env var needed. The trace ID goes into the Serilog enricher — one line: loggerConfig.Enrich.WithProperty("TraceId", Activity.Current?.TraceId) — and into App Insights correlation automatically since the App Insights SDK reads Activity.Current.

Basis System.Diagnostics.ActivitySource + Activity are built into .NET 5+ — no extra NuGet for the API itself. App Insights SDK reads Activity.Current for correlation automatically; no additional wiring needed. FlowlineCommand.ExecuteAsync is the single entry point — one place to start and dispose the root activity. CI detection via standard env vars: CI=true (GitHub Actions, CircleCI), GITHUB_ACTIONS=true, TF_BUILD (Azure DevOps), TEAMCITY_VERSION. Machine identity without PII: Environment.MachineName hashed (stable across runs, not personally identifiable).
Replaces I4 Child activities replace the DiagnosticContext List<string>: each stage is using var stage = _source.StartActivity("PacSync"), disposed when the stage completes. On failure the activity carries status Error and the exception message. No custom type, no global state, no dispose-on-success concern. OTel-aligned: if an OTLP exporter replaces App Insights later, the instrumentation is unchanged.
User & CI tracking for App Insights Set as Activity tags on the root span and as OTel Resource attributes (I7): user.id = stable hash of Environment.MachineName; ci = true/false; ci.platform = "github" / "azuredevops" / "teamcity" / "other"; flowline.version; os = RuntimeInformation.OSDescription. Resource attributes set via ResourceBuilder.AddService().AddAttributes() — no TelemetryInitializer needed. Enables App Insights queries like "show all crashes on CI" or "which machine/user hits this exit code most."
Rationale Consolidates I4 and the original I6 into one standard mechanism. CI pipelines run multiple Flowline commands sequentially — without a shared trace ID, the log files are independent. With TRACEPARENT set from the pipeline run ID, all log files from that run share a W3C-standard identifier and are trivially joinable. Pairs with I3 (log files) and I7 (App Insights) to complete the observability story.
Downsides Stage child activities require a using block or explicit Dispose() at each stage boundary — same mechanical work as I4, but the result is richer. CI operators must know to set TRACEPARENT for cross-command correlation; auto-generated root IDs handle the common case. ActivitySource.StartActivity returns null when no listener is registered — all call sites must null-check or use the ?.SetTag() pattern.
I7.

App Insights Telemetry

Remote Telemetry Confidence 65% Complexity L Planned · Wave 4

On any invocation: send traces and exception events to Azure Application Insights via the OTel OTLP exporter (Azure.Monitor.OpenTelemetry.Exporter). The ActivitySource from I6 feeds directly into an OTel TracerProvider configured with the Azure Monitor exporter — no TelemetryClient, no classic App Insights SDK. Payload: exception type name, exit code enum value, command name (no args), Flowline version, OS/arch, CI flag, hashed machine ID. No PII: no file paths, no Dataverse URLs, no user identifiers. Disclose at first run to stderr. Honor FLOWLINE_TELEMETRY_OPTOUT=1 and DO_NOT_TRACK=1 unconditionally. Consent model (opt-in vs. opt-out default) TBD — see field below. Before process exit: tracerProvider.ForceFlush(TimeSpan.FromSeconds(5)) then dispose — the exporter drains its buffer synchronously.

Stack OpenTelemetry SDK + Azure.Monitor.OpenTelemetry.Exporter. TracerProvider built in Program.cs before app.RunAsync; disposed after. I6's FlowlineActivitySource registered as a source on the provider — zero additional instrumentation. Resource attributes via ResourceBuilder.AddService("flowline", version).AddAttributes([ci, os, user.id]) — these land as custom dimensions in App Insights and are queryable across all telemetry.
Why OTLP over classic SDK No TelemetryClient, no InMemoryChannel, no TelemetryInitializer boilerplate. Exporter is swappable: replacing UseAzureMonitorExporter() with AddOtlpExporter() points the same pipeline at any OTLP-compatible backend (Jaeger, Grafana, self-hosted collector) — instrumentation unchanged. Flush contract is simpler: TracerProvider.ForceFlush() is synchronous and guaranteed; classic SDK's FlushAsync had a 30s/500-event buffer that silently dropped events in sub-second CLI runs.
Basis dotnet CLI's opt-out model (DOTNET_CLI_TELEMETRY_OPTOUT, first-run disclosure, CI detection) is the established .NET precedent. DO_NOT_TRACK=1 convention respected by GitHub CLI and npm. Azure.Monitor.OpenTelemetry.Exporter is Microsoft's recommended OTel-native path for App Insights as of .NET 8+ — the classic SDK is in maintenance mode.
Prior Art — FetchXmlBuilder Jonas Rapp's FetchXmlBuilder (XrmToolBox plugin) uses two hardcoded Azure App Insights keys initialized on plugin load. Tracks: Load, New, Clone, Close, action names, assembly version. No opt-out documented anywhere. Aggregated stats published publicly at jonasr.app/xtb-stats/. Received no meaningful backlash — Power Platform community is small and non-privacy-vigilant; the data collected is minimal and the tool is beloved. False signal for Flowline: Flowline's enterprise target audience is more GDPR-exposed and security-scrutinized than the hobbyist XrmToolBox user base. Silence works in a small community but is fragile — one blog post from an enterprise user can flip the narrative.
Ecosystem Precedents dotnet CLI (DOTNET_CLI_TELEMETRY_OPTOUT): opt-out + first-run disclosure. Received backlash but established the accepted .NET precedent. Linux Mint shipped with it disabled by default.
Go toolchain: proposed opt-out telemetry → massive backlash → backed off entirely.
GitHub CLI v2.91.0 (Apr 2026): silently enabled opt-out with no announcement → significant backlash. Lesson: silent enablement is the cardinal sin, not opt-out itself.
Core principle: opt-out is defensible when disclosure is prominent and immediate. "If telemetry were genuinely beneficial for users, they would be loud about it" — the consent inversion is the ethical problem, not the data volume.
Consent Model — TBD Opt-out (dotnet CLI model): higher telemetry signal early, broader dataset. Risk: enterprise users may object post-hoc; GDPR legitimate-interest argument required.
Opt-in (interactive first-run prompt): explicit consent, no backlash risk, GDPR-clean. Risk: lower adoption rate means weaker signal when you most need it. Can always relax to opt-out once users trust the tool.
Regardless of default: honor FLOWLINE_TELEMETRY_OPTOUT=1 and DO_NOT_TRACK=1 unconditionally. Publish exact field list on the wiki. Consider a flowline telemetry command to inspect and toggle the setting — removes the "hidden surveillance" framing before it starts.
Rationale Local log files only tell you about bugs users report. Telemetry reveals what's actually breaking across all users — the silent majority of crashes that never turn into GitHub issues. For a developer tool in early adoption, aggregate crash data (which commands fail most, which exit codes are hit in the wild, how many runs are CI vs. local) is strategically valuable for prioritization. Ship Waves 1–3 first; telemetry is most valuable once local logging is stable and you want aggregate signal beyond what local logs provide.
Downsides Enterprise egress (major): enterprise GitHub Actions and self-hosted agents may block *.in.applicationinsights.azure.com via L7 firewall. Telemetry must never block or slow the CLI — use a short ForceFlush timeout (5s) and swallow flush failures silently. GDPR/enterprise trust: Flowline targets enterprise Power Platform teams with legal obligations — highest-risk segment for telemetry pushback; corporate security reviews may flag any outbound data collection regardless of what is collected. Complexity: requires Azure App Insights resource, connection string distribution, ongoing maintenance. Consent model undecided — see TBD field above. Lowest confidence (65%) — treat as a separate product decision.
