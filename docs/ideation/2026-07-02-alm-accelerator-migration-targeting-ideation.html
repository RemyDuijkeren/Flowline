Codebase Context

Flowline CLI (current state)

C# .NET 10 CLI, v0.8.0. Three projects: Flowline, Flowline.Core, Flowline.Attributes. Commands: sync, deploy, configure (planned). DTAP gates at DeployCommand.cs:82–97 with non-bypassable DevBlock. Two-phase orphan cleanup via OrphanCleanupService at DeployCommand.cs:61,67. GitUtils.CreateTagAsync exists. SolutionChangeSummary.cs:334–338 parses env var defs from XML; DataverseContextGenerator.cs:565–596 reads connection reference defs. STRATEGY.md explicitly excludes the ISV/source-as-truth model.

ALM Accelerator (deprecated 2025)

Canvas admin app + Azure Pipelines YAML. Key features:

  • Per-env deployment settings JSON (conn refs, env vars, ownership, Teams webhooks)
  • Solution export → ADO branch commit workflow
  • Solution Checker as PR build gate
  • Multi-env promotion with ADO approval gates
  • Maker canvas app persona + admin persona

Required Power Apps per-user license + Azure DevOps Basic.

Official Replacements & Gaps

Pipelines for Power Platform — Microsoft's stated successor. Low-code GUI pipeline. Cannot pack or deploy plugins or web resources. No code-first ALM support.

Dataverse Git Integration — direct Dataverse→Git sync. Competing model (source-as-truth), not PROD-as-truth.

Gap: No official tool does code-first ALM (plugins + web resources + low-code components, PROD-as-truth). Flowline's core differentiator is unchallenged by the official replacements.

Per-env Deployment Settings File

Axis 2 — Per-Env Configuration Conf 95% Complexity Low
Description
Commit flowline.env.{env}.json alongside the solution (e.g. flowline.env.prod.json, flowline.env.test.json). When flowline deploy {env} or flowline configure runs, the matching file is auto-loaded and drives: connection reference resolution (name → connection ID in that env), environment variable values, and owner assignments per environment. No more manual post-deploy configuration in the target environment; unattended CI deploys become possible.
Axis
Axis 2 — Per-Env Configuration
Basis
direct: ALM Accelerator's most-cited feature was its per-env deployment settings JSON — committed to source, loaded by the pipeline, resolving conn refs and env vars per target. SolutionChangeSummary.cs:334–338 already parses env var definitions from solution XML; DataverseContextGenerator.cs:565–596 reads connection reference definitions — parsing infrastructure is present. The 2026-07-01-post-deploy-environment-config-ideation.html confirms this surface is under active design.
Rationale
ALM Accelerator migrants will immediately ask "where does my deployment settings file go?" — this was the feature that made ALM Accelerator worthwhile for multi-environment teams. Without it, every Flowline deploy to Test/UAT/Prod requires a developer to manually configure connection references and env vars in the target, breaking unattended CI. The parsing infrastructure already exists; this is the highest value-per-effort item for migration targeting.
Downsides
Secure values (passwords, API keys) must not be committed — the file stores references to Key Vault secrets or env variable placeholder names, not secrets themselves. File schema must be versioned carefully; a breaking change silently breaks all deploys. Needs clear documentation on what is and is not safe to commit.

ALM Accelerator Migration Guide + Comparison Doc

Axis 5 — Migration Targeting Conf 95% Complexity Low
Description
A docs/others/alm-accelerator-comparison.md following the 471-line spkl-comparison.md template, plus a wiki/15-Migration-from-ALM-Accelerator.md page. Covers: what ALM Accelerator did, which parts Flowline replaces, which parts the official replacements (Pipelines for Power Platform, Dataverse Git Integration) handle, and which parts fall away as unnecessary in Flowline's PROD-as-truth model. Migration guide gives a concrete two-phase path: phase 1 (adopt Flowline for code-first components alongside existing ALM Acc pipeline) → phase 2 (drop ALM Acc, adopt per-env settings file and flowline ci init templates).
Axis
Axis 5 — Migration Targeting
Basis
direct: docs/others/spkl-comparison.md (471 lines) establishes the template. wiki/12-Migration-from-spkl.md, wiki/13-Migration-from-Daxif.md, wiki/14-Migration-from-PACX.md confirm the two-phase migration pattern as an established wiki convention. ALM Accelerator deprecated 2025 creates an active cohort searching for alternatives right now.
Rationale
Highest value-per-hour item in this set — it's documentation, not code, and the template already exists. ALM Accelerator's deprecation is a timed event producing search traffic and forum questions ("what do I use instead?") that a clear comparison doc captures. The two-phase migration respects teams that can't migrate all at once, lowering the adoption barrier. Builds on three prior migration guides already in the wiki.
Downsides
Requires ongoing maintenance as ALM Accelerator officially sunsets and the official replacements mature. Risk of overpromising: features not yet shipped in Flowline (e.g. per-env settings file) must be clearly flagged as planned, not present — recommending migration before the features exist undermines trust.

flowline status --compare <env1> <env2>

Axis 4 — CI/CD Quality Gates Conf 85% Complexity Medium
Description
Export the solution from both <env1> and <env2>, unpack both, run SolutionChangeSummary across the pair, and output a component-level diff table — which components exist in one env but not the other, which are at different versions, which have drifted. Usable standalone ("show me what Test is missing vs Prod") or as a CI quality gate ("Test is not more than N components behind Prod"). Output is structured text; --format json for CI parsing.
Axis
Axis 4 — CI/CD Quality Gates
Basis
direct: docs/others/BeforeReleaseV1.md:20: "env-to-env comparison: This is what will drive word of mouth — developers will demo this to each other because nothing else does it." SolutionChangeSummary.cs already parses component-level diffs from unpacked solution XML — the mechanism is a direct extension of existing code. No other tool (ALM Accelerator, Pipelines for Power Platform, PAC CLI) provides a component-level env diff.
Rationale
BeforeReleaseV1.md calls this the word-of-mouth driver — nobody else does it. ALM Accelerator offered no equivalent; nor do Pipelines for Power Platform or PAC CLI. The mechanism extends code that already exists. Demonstrating it in a meeting — "here's what Test is missing compared to Prod" — is immediately compelling to any team managing multi-environment deployments.
Downsides
Requires export credentials for both environments — developer or CI must authenticate to both. Export of a large solution takes time; the command can be slow. Component-level diff on solution XML requires robust handling of ordering differences that are not semantic changes (XML attribute order, whitespace normalization).

flowline ci init --github / --ado

Axis 5 — Migration Targeting Conf 90% Complexity Medium
Description
Generate a working, copy-paste-ready CI/CD YAML from the project's .flowline config. --github writes .github/workflows/flowline-deploy.yml; --ado writes azure-pipelines.yml. The generated YAML calls flowline sync, flowline check, and flowline deploy {env} with solution name and env names pre-filled from .flowline. Includes stubs for the git-tag approval gate (idea #6) and solution checker step (idea #5). ALM Accelerator migrants coming from Azure Pipelines YAML get a replacement pipeline in under a minute.
Axis
Axis 5 — Migration Targeting
Basis
direct: docs/others/BeforeReleaseV1.md:25–27: "GitHub Actions / Azure DevOps example pipelines. Not a feature — just documentation. But critical for the CI/CD story… copy-paste ready." A generator reads the actual .flowline config and pre-fills names, eliminating copy-paste error risk. ALM Accelerator's core value was largely its Azure Pipelines YAML templates; migrants understand YAML-based CI and will reach for this immediately.
Rationale
BeforeReleaseV1.md names CI/CD templates as "critical." A generated version that reads .flowline and pre-fills names is strictly better than static docs. For ALM Accelerator migrants, replacing the Azure Pipelines YAML is the first concrete step after installing Flowline. Composable with ideas #5 and #6 — the generated YAML includes stubs for both, making adoption of those features a one-line uncomment.
Downsides
Generated YAML must be kept in sync with Flowline's CLI surface — a breaking CLI change silently fails pipelines in repos that haven't regenerated. Must support at least GitHub Actions and Azure DevOps to cover the ALM Accelerator audience; each additional platform adds maintenance scope.

Solution Checker Gate (flowline check)

Axis 4 — CI/CD Quality Gates Conf 85% Complexity Low
Description
A standalone flowline check command that runs pac solution check against the zip artifact already produced by PackSolutionAsync at DeployCommand.cs:210. Returns ExitCode.ValidationFailed=15 on errors — a stable, named CI signal. Usable standalone in a CI pipeline step before deploy, or as a pre-deploy gate wired to the existing pack step. ALM Accelerator ran solution checker in PR builds; this surfaces the same quality signal without requiring ALM Acc infrastructure.
Axis
Axis 4 — CI/CD Quality Gates
Basis
direct: PackSolutionAsync at DeployCommand.cs:210 already produces the zip artifact — no extra export step needed. ExitCodes.ValidationFailed=15 is a stable, named exit code designed for CI parsing. pac solution check is already in the PAC CLI dependency chain. ALM Accelerator's solution checker gate was one of its most-valued CI features, cited repeatedly in migration discussions.
Rationale
Low implementation burden: the zip artifact exists and PAC CLI is already a dependency. ExitCode=15 makes results immediately parseable by any CI system. Running solution checker in CI was an ALM Accelerator best practice that many teams relied on; surfacing it as a first-class Flowline command closes that gap with minimal new code.
Downsides
pac solution check requires internet connectivity and a Power Platform environment; it cannot run fully offline. False positives on third-party managed components are common — Flowline may need a suppression list for known-false-positive rules. PAC CLI result format may change across versions.

Git Tag as Approval Gate for Prod Deploy

Axis 1 — DTAP Gate Enforcement Conf 85% Complexity Low
Description
Require a Git tag matching approved/uat/v{version} before flowline deploy prod proceeds. The approval action is "push the tag" — no external service, no approval portal. On successful prod deploy, Flowline auto-creates a receipt tag receipt/prod/v{version} so the audit trail lives in Git. The gate hooks into ValidateDtapGateAsync at DeployCommand.cs:132; tag creation reuses GitUtils.CreateTagAsync.
Axis
Axis 1 — DTAP Gate Enforcement
Basis
direct: ValidateDtapGateAsync at DeployCommand.cs:132 is the exact hook — DTAP gate validation is already a function call that can be extended to check tags. GitUtils.CreateTagAsync exists in src/Flowline/Utils/GitUtils.cs. reasoned: Push-the-tag is a proven GitOps approval pattern (used in Helm, ArgoCD, Flux) that puts approval in Git history, making the audit trail durable and free. ALM Accelerator used ADO PR status as a gate — this achieves the same intent with zero external service dependency.
Rationale
ALM Accelerator's approval workflow required ADO pull request status — a dependency GitHub-based teams couldn't use. A git tag requires no external service, works with any CI system, and leaves a durable audit record. "Push a tag to approve" is two commands and is readily scriptable. The receipt tag closes the loop: git tag -l "receipt/prod/*" shows every successful prod deploy, no log scraping required.
Downsides
Anyone with push access can push the approval tag — it's an honor system without additional branch protection rules. Teams used to UI-based approval flows will find this more manual. Version string parsing must be robust — mismatches between tag format and .flowline config version will silently block deploys.

Post-deploy Flow Ownership Assignment

Axis 3 — Post-Deploy Lifecycle Conf 80% Complexity Low–Med
Description
An owners section in the per-env settings file (idea #1) that maps flow names to owner user or team IDs for the target environment. flowline configure issues UpdateRequest on the corresponding workflow Dataverse records after deploy, setting ownership to the declared owner. ALM Accelerator's deployment settings included an ownershipTable for flows — this is the Flowline equivalent.
Axis
Axis 3 — Post-Deploy Lifecycle
Basis
direct: OrphanCleanupService.cs already issues UpdateRequest on workflow records — the Dataverse update mechanism is proven and in production. Hook points at DeployCommand.cs:61,67 (pre/post import) show the established pattern for post-deploy Dataverse actions. ALM Accelerator's deployment settings explicitly included an ownershipTable section for flows. Note: this is NOT covered by the 2026-07-01-post-deploy-environment-config-ideation.html (which covers flow activation, connection assignment, env var setting — not ownership assignment).
Rationale
Deployed flows in a fresh environment default to the deploying user as owner. On shared Test/Prod environments this creates ownership drift: when the deploying user's credentials expire or the user leaves, flows fail. ALM Accelerator solved this explicitly. The update mechanism already exists in OrphanCleanupService; this extends an existing pattern rather than introducing new infrastructure. Depends on idea #1 (per-env settings file) as the natural config location.
Downsides
Owner IDs are environment-specific — the settings file must store IDs valid in the target env, not portable display names. If a named owner doesn't exist in the target, the update fails silently unless error handling is explicit. Not independently shippable without idea #1 as the config mechanism.
