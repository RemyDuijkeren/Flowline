Ideation: Verbose Log Routing

Focus: Always send verbose-level output to the log file, even when --verbose is not set. Options for decoupling verbose capture from verbose display — including reversing the model: always generate verbose internally, filter before the terminal.

Phase 1

Grounding Context

What Exists Now

  • Serilog at MinimumLevel.Debug, file-only sink — would capture everything if content reached ILogger
  • LoggingRenderHook intercepts every Spectre Markup/Tree/Panel/Table → ILogger → file
  • VerboseOutputBuffer accumulates context on error path
  • Console.Verbose(msg, options): IsVerbose=true → MarkupLine → hook → file
  • WithToolExecutionLog(options): IsVerbose=true → stdout → MarkupLine → hook → file

Current Gaps

Gap A · Flowline's own verbose messages
  • Console.Verbose(msg, options) when !IsVerbose: buffer only — zero ILogger write
  • Console.Verbose(msg, bool) when false: nothing at all — no buffer, no file
Gap B · External subprocess stdout
  • WithToolExecutionLog(options) when !IsVerbose: non-error stdout silently dropped
  • WithToolExecutionLog(bool) overload: no buffer at all — stdout dropped, zero ILogger, zero buffer regardless of flag
  • PacUtils buffered calls (ExecuteBufferedAsync): stdout goes to result.StandardOutput for JSON parsing — pipe delegate never sees it at all

Shared consequence: VerboseOutputBuffer flushed to ILogger only on error — verbose context lost on success for both gaps.

External Prior Art

  • Azure CLI: always writes full debug to timestamped files regardless of --debug/--verbose; terminal gated by flag — production-validated
  • MSBuild: /bl binary log always at diagnostic verbosity; console verbosity set independently
  • Serilog LoggingLevelSwitch: runtime-adjustable level per sink, post-initialization

Key Structural Constraint

  • No Serilog console sink exists — terminal output is Spectre-native. Ideas requiring a Serilog console sink to filter on were rejected.
  • ILogger is not reachable from FlowlineConsoleExtensions (static extension with no DI). The FlowlineRuntimeOptions parameter pattern is the established workaround.
  • LoggingRenderHook only sees content that flows through Spectre's MarkupLine — anything bypassing it is invisible to the file.

1.
Generate Always, Gate at Terminal
Axis 1 · Decoupling point Conf · High Cx · XS Gap A principle Gap B principle

The architectural principle: the --verbose flag must never gate whether content is generated — only whether the terminal displays it. Every verbose message exists at the ILogger layer unconditionally; the terminal is a display preference, not a capture filter. This is the decision that ideas 2 and 3 implement. Its value is in being named and enforced rather than improvised per call site.

Before: --verbose gates content generation (capture filter). After: ILogger always receives the event; terminal is gated separately (display filter).

Basis direct: Console.Verbose(msg, bool) line 21-25 — if (isVerbose) MarkupLine else nothing direct: Console.Verbose(msg, options) line 27-33 — if (IsVerbose) MarkupLine else VerboseOutput.Append (no ILogger) external: Azure CLI always writes full debug to file regardless of flag; terminal is a display-layer decision
Rationale The file sink is at MinimumLevel.Debug — it would capture verbose content unconditionally if that content reached ILogger. The only obstacle is the flag gates in the call sites. Moving the gate downstream makes the log file unconditionally complete. Ideas 2 and 3 are the implementation; this is the principle that names the decision and prevents future regressions.
Downsides Not a code change on its own — needs idea 2 and/or 3 to implement. Value is in naming and enforcing the rule (e.g., as an architectural comment in LoggingRenderHook or Program.cs).

The --verbose flag is currently a capture filter. It decides whether the event is generated, not whether the terminal displays it. The fix is to move it to a display filter. Should we audit every verbose call site for this?

2.
Direct ILogger Injection in Console.Verbose()
Axis 3 · File routing Conf · High Cx · S Gap A ✓

Add an ILogger parameter to both Console.Verbose() overloads and call logger.Debug(message) unconditionally — before the terminal-display branch. The call always reaches the Serilog file sink regardless of --verbose. The terminal branch (MarkupLine or buffer) remains as-is; this is purely additive. The parameter injection pattern mirrors how FlowlineRuntimeOptions is already threaded through.

Basis direct: FlowlineConsoleExtensions.cs:27 — options parameter already threaded; ILogger can follow same pattern direct: MinimumLevel.Debug() set in Program.cs:72 — ILogger.Debug() WILL reach the file direct: LoggingRenderHook only fires on MarkupLine — content suppressed from Spectre is invisible to the file without a direct ILogger path
Rationale The Serilog file is ready; the only missing piece is the ILogger call itself. Every call site already passes FlowlineRuntimeOptions — adding ILogger alongside it is the minimal change. The hook's complexity stays scoped to user-facing output where it belongs; verbose content gets a direct, simple route to the file.
Downsides Requires adding ILogger parameter to both verbose overloads + updating all call sites. Static extension methods have no DI — callers must pass ILogger explicitly. The bool overload (line 21) should probably be removed rather than extended, since it has no buffer path either.

Both verbose overloads are structurally blocked from reaching ILogger. The fix is one parameter and one unconditional logger.Debug() call. Should we remove the bool overload and consolidate on the options overload?

3.
Always-Capture Subprocess Stdout via ILogger
Axis 3 · File routing Conf · High Cx · S Gap B ✓

Change WithToolExecutionLog's non-verbose stdout delegate to always pipe each line to ILogger.Debug(s) before discarding it from the terminal. Currently when IsVerbose=false, the delegate does nothing with non-error stdout — it is silently dropped. Adding one logger.Debug(s) call inside that delegate closes the subprocess capture gap: PAC, dotnet, and git output always appear in the log file, regardless of the flag.

Basis direct: CommandExtensions.cs:96-102 — non-verbose stdout delegate: .WithStandardOutputPipe(PipeTarget.ToDelegate(s => { SetStatusWithExecutionTime(ctx, s); if (IsErrorLine(s)) options.VerboseOutput.Append(s); // ← non-error stdout is DROPPED here })) external: Azure CLI pipes all tool output to debug file unconditionally; terminal display is the only thing gated
Rationale Subprocess output (PAC import counts, entity names, dotnet build lines, git status) is the most valuable diagnostic signal during a failed deployment. It vanishes entirely unless the operator ran with --verbose. One logger.Debug(s) call in the non-verbose delegate permanently closes this gap — all current and future subprocess calls gain unconditional file capture.
Downsides Requires ILogger to be available inside the delegate. CommandExtensions is currently static with no ILogger parameter. Same solution as idea 2: thread ILogger through the options object or add it as a parameter. Stdout from long-running PAC operations can be voluminous — log file size may increase meaningfully on verbose operations.

A PAC solution publish succeeds but registers the wrong version. The log file has no stdout from PAC because we weren't running --verbose. Should subprocess stdout always be in the log file?

4.
Flush Buffer to ILogger on All Exits
Axis 4 · Buffer future Conf · High Cx · XS Gap A partial Gap B partial

The simplest standalone fix: call FlushBufferedVerboseOutput (or equivalent) after app.RunAsync() on the success path too, not only from the exception handler. Currently the success path is just RunAsync → CloseAndFlush → Dispose → return with no buffer flush. Adding one flush call means whatever is buffered (the last ≤50 verbose messages via the options overload) always reaches the Serilog file at the end of every run. This doesn't depend on ideas 2 or 3 and can ship independently.

Basis direct: Program.cs:184-186 — success path: var exitCode = await app.RunAsync(...); Log.CloseAndFlush(); return exitCode; // ← no buffer flush before CloseAndFlush direct: FlushBufferedVerboseOutput(Exception) called only from exception handler (lines 104, 113)
Rationale The buffer exists to collect verbose context. The file sink is the right place for that context. On success, the buffer is currently garbage-collected with its contents. One call before CloseAndFlush fixes this. The change is three lines. It doesn't require any other structural change and closes the success-path gap for all content already being buffered.
Downsides Partial fix only: covers only what's already in the buffer. The bool overload verbose path (idea 2) and subprocess stdout drops (idea 3) are still not captured. Still limited to 50 lines. Buffer on success should NOT re-print to terminal (the FlushBufferedVerboseOutput function does terminal output too — need a file-only variant).

The buffer collects verbose context but only writes it to the file when something throws. Should we always flush it to the log file at run end, regardless of success or failure?

5.
Separate VerboseFilterHook (the Reverse Model, Refined)
Axis 2 · Terminal suppression Conf · Med Cx · M Gap A ✓

The reverse model refined: instead of giving LoggingRenderHook two jobs (log + suppress), split into two hooks registered in sequence. LoggingRenderHook stays unchanged — it already logs every Markup renderable to ILogger unconditionally. A new VerboseFilterHook, registered second in the Spectre pipeline, intercepts VerboseMarkup renderables and suppresses their yield when !IsVerbose. Call sites use VerboseMarkup (a wrapper type extending Markup) instead of raw Markup("[dim]..."). LoggingRenderHook sees it first (logs it), then VerboseFilterHook decides whether the terminal gets it. The [dim]/Skip() ambiguity is resolved by type — Skip() still uses plain Markup, so the filter never touches it.

Hook order: LoggingRenderHook (registered first) always logs to file. VerboseFilterHook (registered second) suppresses terminal display of VerboseMarkup when !IsVerbose. Skip() uses plain Markup — filter never sees it.

Basis direct: LoggingRenderHook.Process line 17 — "if (renderable is Markup...)" catches VerboseMarkup too (subtype); already logs to ILogger unconditionally when IsVerbose=true direct: LoggingRenderHook line 13 — "yield return renderable" always passes to next hook in the pipeline; a second hook can then suppress direct: FlowlineConsoleExtensions.cs:19 — Skip() uses plain Markup("[dim]..."), not a subtype; type-based filter would never match it direct: Program.cs line 180 — "AnsiConsole.Console.Pipeline.Attach(...)" — hooks are in a list; registration order determines pipeline order
Rationale Single responsibility: LoggingRenderHook does logging (it already does), VerboseFilterHook does terminal suppression. Neither hook needs to know about the other's job. LoggingRenderHook requires zero changes. VerboseFilterHook is clean: one check, one decision. The [dim] ambiguity with Skip() is solved structurally by type — no string prefix scanning or convention required. Adding new verbose output never requires an IsVerbose check at the call site.
Downsides Call sites must change from Markup("[dim]...") to new VerboseMarkup("...") — a broad but mechanical refactor of all Console.Verbose() call sites.
VerboseFilterHook needs FlowlineRuntimeOptions.IsVerbose at runtime — new runtime dependency on a DI object that hooks don't currently hold.
Subprocess stdout via WithToolExecutionLog never reaches Spectre when !IsVerbose — hook pipeline never fires for it. Still needs idea 3 (or idea 7) to capture subprocess output.

LoggingRenderHook stays unchanged. VerboseFilterHook is new and small. Call sites switch to VerboseMarkup — one mechanical pass. Subprocess capture is the remaining gap. Is that the direction?

6.
Reverse the Default: --quiet Instead of --verbose
Axis 2 · Terminal suppression Conf · Low Cx · L Gap A+B terminal only

Flip the assumption: verbose output is the default; --quiet suppresses it. Operators who want clean CI output pass --quiet. Local developers who forget a flag get full diagnostic context immediately, not a terse error that requires a re-run. The penalty for forgetting a flag drops from "permanently lost context" to "extra terminal lines." This is a separate UX direction from fixing the file log — it addresses what the terminal shows by default.

Basis external: Azure CLI — default output is full; --only-show-errors suppresses; --debug adds more reasoned: --verbose has asymmetric cost: forgetting it on failure means re-running an entire PAC operation (minutes). Forgetting --quiet costs only terminal noise. Penalize the cheaper mistake.
Rationale A developer's first failing flowline deploy should show full context — not require discovering and re-running with --verbose. The user who wants clean output is more sophisticated and will find --quiet; the user who needs context is often new and will forget --verbose.
Downsides Breaking change to CLI interface — all scripts, docs, and habits that use --verbose would need updating. The file-log gap argument weakens once ideas 2+3 ship (file is then always complete). Remains primarily a terminal UX decision with a distinct value proposition from file completeness. Separate brainstorm territory.

Should verbose output be on by default and --quiet be the opt-out? The penalty for forgetting --verbose is worse than forgetting --quiet.

7.
Replace CommandExtensions with SubprocessCapture
Axis 5 · Subprocess architecture Conf · Med Cx · M–L Gap B ✓

The static extension design of CommandExtensions is what makes ILogger unreachable — static methods have no DI. Replace both WithToolExecutionLog overloads with a SubprocessCapture class that takes ILogger and FlowlineRuntimeOptions in its constructor. Every stdout line is piped to ILogger.Debug() unconditionally; terminal display is a separate decision. Error detection, PAC async progress filtering, line transforms, and status context updates — currently duplicated inside the two overloads — are consolidated in one class. PacUtils and other callers get SubprocessCapture via DI, ending the bool verbose parameter threading across half the codebase.

Before: static extensions gate on bool verbose; no ILogger access. After: SubprocessCapture holds ILogger, always writes to file, terminal display is conditional. Two subprocess patterns (streaming/buffered) need distinct handling.

Basis direct: CommandExtensions.cs:8 — "public static class CommandExtensions" — static, no DI, no ILogger direct: CommandExtensions.cs:96-103 — non-verbose path: stdout dropped except error lines buffered; stderr buffered only; zero ILogger calls direct: PacUtils.cs — GetEnvironmentsAsync/GetSolutionsAsync/GetSolutionVersionAsync use .WithToolExecutionLog(verbose) with ExecuteBufferedAsync — these are buffered calls; stdout goes to result.StandardOutput for JSON parsing, not to a pipe delegate; the pipe only serves display direct: PacUtils.cs:184-197 — PackSolutionAsync uses WithToolExecutionLog(verbose, ctx) with streaming ExecuteAsync — genuine streaming, all stdout flows through delegates direct: CommandExtensions.cs:19-57 vs 60-103 — two overloads (bool, options) with largely duplicated logic; SetStatusWithExecutionTime, IsErrorLine, DisplayErrorMessage, lineTransform gating all appear twice
Rationale Static extensions can't hold state. The ILogger gap is an architectural consequence of the design, not a fixable oversight. A DI-injectable class ends the problem class-wide: every subprocess invocation — present and future — automatically logs to file without an explicit ILogger parameter thread. The deduplication of the two overloads is a side benefit. Replacing a utility class is narrower in scope than it looks: CommandExtensions is used primarily from PacUtils and a handful of commands.
Downsides Two subprocess patterns (streaming vs buffered) need different APIs. The buffered calls in PacUtils don't use pipe delegates for display at all — stdout goes to result.StandardOutput for JSON parsing. A SubprocessCapture can't intercept that without wrapping the buffered call itself, which changes the return type.
PacUtils is currently static (public static class PacUtils) — injecting SubprocessCapture requires either making PacUtils non-static, adding SubprocessCapture as a parameter to static methods, or splitting PacUtils into static helpers (pure functions) and an injectable facade.
Larger refactor scope than ideas 2+3: touches PacUtils.cs (~460 lines), CommandExtensions.cs, and all command callers. Risk of breaking the streaming/buffered contract during migration.

CommandExtensions can't hold ILogger because it's static. A SubprocessCapture class with ILogger in the constructor permanently solves the gap for all current and future subprocess calls. But PacUtils is also static — how far does the refactor reach?
