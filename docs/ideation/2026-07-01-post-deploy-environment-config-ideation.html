Classic Workflow State Restoration

Flow & workflow lifecycle Conf 90% Complexity: Low

Description

Solution import silently resets classic workflows and BPFs to Draft state — every deployment to any non-dev environment currently breaks them without error. Flowline takes a pre-import snapshot of workflow.statecode for all components in the solution, then post-import restores them via SetStateRequest. The developer never manually re-activates workflows again. A --no-restore flag provides an opt-out.

Basis

direct: Brainstorm 2026-06-12-deploy-state-restoration-requirements.md covers exactly this scenario (status: idea). OrphanCleanupService.cs already calls UpdateAsync with statecode/statuscode for workflow deactivation before deletion — the SDK pattern is proven in the codebase. Post-import hook at DeployCommand.cs:67 is the insertion point.

Rationale

The pattern (pre-import snapshot → post-import restore) is structurally identical to how OrphanCleanupService works. This is the second service following the same shape — no new architecture. The existing brainstorm already scoped it; this validates it as the right next implementation candidate.

Downsides

Restoring a workflow that was deactivated intentionally in the target environment — by a support engineer or admin outside of Flowline — would silently undo that. The --no-restore flag handles opt-out, but drift detection (Idea 6) is the fuller answer.

Step State Declaration Drives Deploy

Plugin step state & config Conf 88% Complexity: Low

Description

Add IsDisabled to StepAttribute, then honor it on flowline deploy — not just on push. After import, Flowline calls SetStateRequest on sdkmessageprocessingstep to match declared state. The attribute becomes authoritative across all Flowline operations: push registers the step with correct state, deploy reconciles it back after import. No separate command, no post-deploy checklist item.

Basis

direct: StepAttribute.cs currently has Order, RunAs, Config, Description, SecondaryTable, DeleteJobOnSuccess — no IsDisabled. The codebase scan confirmed this property is missing; adding it is a small addition. SetStateRequest on sdkmessageprocessingstep (State 0 = Active, State 1 = Inactive) is a standard SDK call. PAC --activate-plugins is all-or-nothing with no per-step selectivity — there's a confirmed gap.

Rationale

Closing the push/deploy gap makes [Step] the single source of truth for step behavior. Every other Flowline feature reasons from declared attribute state — step enablement should be no different. Once IsDisabled exists in the attribute, per-env override support (step X disabled in UAT, active in prod) becomes a natural extension.

Downsides

Requires adding IsDisabled to StepAttribute first (not present today). If a developer marks a step disabled and forgets it when debugging in prod, the next deploy quietly re-disables it. Strong convention: the attribute is authoritative.

Precondition Add IsDisabled to StepAttribute — a trivial one-property change — before honoring it in the deploy pipeline.

Pre-Flight Config Validation

Settings & env vars Conf 85% Complexity: Low

Description

Before issuing pac solution import, Flowline validates that all environment variables and connection references required by the solution exist and are configured in the target environment. If anything is missing, deploy aborts with a clear actionable list: "Deploy blocked: 2 env vars missing values, 1 connection reference unconfigured." No partial import, no silent runtime failure on the first user action in prod.

Basis

direct: DeployCommand.cs already runs ValidateTargetAsync and ValidateDtapGateAsync before import — a validation checkpoint exists. SolutionChangeSummary.cs:334-338 already parses env var definition files from solution XML, establishing the file-reading capability. Solution XML in Package/src/ contains connection reference and env var definitions — readable before import.

Rationale

A failed import wastes 2–10 minutes and can leave the environment in a partially-updated state. Pre-flight makes deploy deterministic: it either succeeds completely or fails cleanly before touching the environment. This is the safety net that makes every other config feature trustworthy.

Downsides

Requires querying the target environment before import — adds a round-trip. --force flag should bypass as an escape hatch when the environment is intentionally partially configured.

Secure Config Deploy-Time Injection

Plugin step state & config Conf 80% Complexity: Medium

Description

Flowline reads a per-environment gitignored file — e.g. .flowline/secure-config.prod.json — and upserts sdkmessageprocessingstepsecureconfig records via SDK post-import. Webhook secrets, API keys, and signing certs get applied automatically as part of flowline deploy prod. No admin console, no manual post-deploy ritual, no bespoke per-project C# script.

The gitignored file pattern keeps secrets out of source control (honoring Flowline's design stance) while making the deployment reproducible. The format mirrors the step name already known to Flowline from registration: { "MyNamespace.StepName: Create of account": "secret-value" }.

Basis

direct: sdkmessageprocessingstepsecureconfig is a separate Dataverse table, not exported in solutions, admin-only read, must be set via API per deployment — confirmed by community research and Microsoft Learn. Current community workaround is bespoke C# scripts per project. external: ALM Accelerator (deprecated 2023) addressed this with sdkstep.sec.<stepid> in its deployment settings JSON — the most complete prior art.

Rationale

Secure config is the most cited reason Power Platform solutions "deploy fine, break in prod." Every plugin step that uses secure config is a silent manual dependency on every environment setup. The gitignored file pattern is CI/CD-native — secrets live in CI environment variables or a vault, the per-env file is generated at deploy time, never committed.

Downsides

Flowline handles secrets during the deploy step — requires careful handling and clear documentation that the file must be gitignored. Existing Flowline design stance explicitly says secure config "is intentionally not supported" (in reference to storing it in the unsecure attribute). This idea works differently (API call, not in source), but the stance may need to be updated.

Design stance note The prohibition in StepAttribute.Config is about unsecure config carrying secrets visible in solution XML — not about calling the secure config API post-deploy. This idea doesn't violate that stance, but the docs should be updated to clarify the distinction.

Ordered Flow Activation

Flow & workflow lifecycle Conf 75% Complexity: Medium

Description

All cloud flows deactivate on import. Activating them in arbitrary order causes silent failures when a parent flow calls a child that's still in Draft state. Flowline activates flows in dependency order — either inferred from flow dependency graphs in the solution XML (clientdata field, implementation risk flagged) or declared explicitly in a config block. Either approach calls workflow.statecode PATCH post-import, activating leaf flows first.

The declared config approach avoids the XML-parsing risk and aligns better with Flowline's convention-over-config philosophy — a short flowline.flows section in .flowline lists activation order and ownership per environment.

Basis

external: ALM Accelerator (deprecated 2023) documented activateflow.activate + activateflow.order + activateflow.activateas for exactly this. Community pain is well-evidenced: "connection reference was configured, flow still off after import" is a repeated forum pattern. The problem is confirmed by the flow-dependency ordering failure mode. direct: DataverseContextGenerator.cs:565-596 already reads connection references from solution XML — the XML-reading infrastructure exists.

Rationale

The first deploy to UAT or prod after adding a new flow often fails on the first customer trigger — not during deploy. Debug time is 20–40 minutes just diagnosing "child flow is Draft." Ordered activation eliminates this class of silent failures. Flow ownership reset (flows running as the wrong user post-import) is a natural companion feature.

Downsides

The clientdata JSON flow-dependency parsing approach is unverified — it's a promising implementation path but requires a real solution XML sample to confirm reliability. The declared config approach is safer but requires user-maintained ordering config. Start with declared, add inferred as a follow-on.

Implementation risk The clientdata-based inferred dependency graph is unverified against real solution XML. Recommend starting with declared ordering in config and treating inferred as a Phase 2 capability once parsing is confirmed.

IPostDeployService Protocol

Integration model Conf 85% Complexity: Medium

Description

Formalize OrphanCleanupService's ad-hoc pre/post import hooks into a typed IPostDeployService interface registered via DI. Each new capability (workflow restoration, step state, flow activation, secure config) becomes a service implementing that interface — the deploy command never changes. Each service is independently testable and shippable.

Basis

direct: DeployCommand.cs already calls orphanCleanupService.RunPreImportAsync() and RunPostImportAsync() — the pattern exists as ad-hoc calls. Formalizing it as a typed interface is the meta-investment that prevents each new capability from branching deploy command logic. The grounding notes "adding env config adds another chain link" — this makes the chain itself a protocol.

Rationale

Without this, every feature (workflow restore, step state, flow activation, secure config) modifies DeployCommand and creates N merge points. With it, DeployCommand stays stable, each capability is independently tested and shippable, and the order of execution is explicit. This is the leverage investment that makes ideas 1, 2, 4, and 5 all cheaper to build.

Downsides

Small refactor required to pull OrphanCleanupService's existing calls behind the interface. Adds indirection — must be worth it when more than 2 services exist. Build this alongside the second service implementation, not in isolation.

Commission as Separate Command

Integration model Conf 80% Complexity: Medium

Description

flowline deploy handles installation: pre-flight validation, pack, import, orphan cleanup, step state reconciliation. A separate flowline commission command handles environment activation: restoring workflow state, activating cloud flows in dependency order, injecting secure config. The distinction is structural — deploy makes the solution present; commission makes it live.

Commission is idempotent and independently re-runnable. When a support engineer manually disables a flow in prod and you want to restore declared state, you run flowline commission prod, not re-deploy. No downtime, no re-import, no risk of overwriting customizations.

Ownership split

Deploy owns

  • Pre-flight config validation
  • Pack solution
  • Import solution (PAC CLI)
  • Orphan cleanup
  • Step state (attribute-driven, deterministic)

Commission owns

  • Classic workflow state restoration
  • Cloud flow activation (ordered)
  • Flow ownership assignment
  • Secure config injection

Default behaviour & flags

# Default: commission runs automatically after deploy
flowline deploy prod

# Split: deploy only (CI step 1)
flowline deploy prod --no-commission

# Commission only (CI step 2, or re-run after manual drift)
flowline commission prod

# Dry-run: show what commission would change
flowline commission prod --dry-run

Why split matters — CI/CD pattern

In a gated pipeline, deploy and commission may run at different stages with different approvals. Deploy runs on every merged PR (import to UAT, automated). Commission runs after human sign-off before going live in prod. The secret for vault-sourced secure config only needs to be present in the commission stage — deploy can remain secret-free.

In regulated environments this maps directly to change-management roles: the DevOps engineer runs deploy; the solution architect or functional lead runs commission to confirm that activation is intentional.

Basis

reasoned: In civil and industrial engineering, commissioning is explicitly distinct from installation — it is the act of verifying a system is operational in its environment, not just physically present. The analogy holds: deploy makes the solution present in Dataverse; commission makes its components run. The Dataverse platform itself echoes this — a plugin step can be registered but disabled; a flow can be imported but deactivated. These are two different states that map to two different owner actions. direct: The brainstorm 2026-06-12-environment-config-command-requirements.md proposes a flowline config command — commission is that direction with a tighter scope and a name that emphasizes activation rather than configuration.

Naming

commission is precise but engineering-domain terminology — unfamiliar to non-technical admins. Alternatives: flowline activate prod (immediate intent, slightly overloaded in Power Platform), flowline configure prod (too broad — could mean env vars), flowline reconcile prod (accurate but jargon-heavy). commission or activate are the two strongest candidates; user research or a naming spike should settle it before shipping the wiki entry.

Downsides

Two-step mental model for users who just want flowline deploy prod to "make everything work." The default auto-commission behavior solves this for most users; the split only matters in CI or regulated contexts. Risk: if commission is optional, users may run deploy without realizing commission is what makes flows go live — the default auto-run must be obvious from the output.

Ad-Hoc State Toggle

Operations Conf 87% Complexity: Low

Description

A flowline toggle command enables or disables individual Dataverse components by name — cloud flows, classic workflows, plugin steps — without re-deploying. Works at any time, against any environment, independently of the deploy/commission cycle. This is the Flowline primitive for the feature-flag pattern: ship dark, enable when ready, roll back instantly.

Env vars are also a target: set or clear a value per-environment to gate features that read from env vars. Secure config injection for a single step can be triggered standalone, without a full commission run.

Command surface

# Enable / disable a cloud flow by display name
flowline toggle flow "New Checkout Flow" --on --env prod
flowline toggle flow "New Checkout Flow" --off --env prod

# Enable / disable a classic workflow or BPF
flowline toggle workflow "Order Approval" --on --env uat

# Enable / disable a plugin step by its registered name
flowline toggle step "MyNs.NewFeaturePlugin: Create of account" --off --env prod

# Set an env var value (feature gate via env var)
flowline set envvar cr123_EnableNewCheckout true --env prod

# Preview what would change (no write)
flowline toggle flow "New Checkout Flow" --on --dry-run --env prod

What each toggle calls under the hood

Cloud flow / classic workflow: workflow.statecode PATCH — 1 = Activated, 0 = Draft. Same SDK call as commission's workflow restore, scoped to one record by name lookup.

Plugin step: SetStateRequest on sdkmessageprocessingstep — State 0 = Active, State 1 = Inactive. Flowline already knows the step's registered name from the assembly scan.

Env var: environmentvariablevalue upsert — create or update the per-environment value record. If the value record doesn't exist yet, create it; if it does, update it.

Secure config (single step): sdkmessageprocessingstepsecureconfig upsert — same mechanism as Idea #4, but targeted at one step without a full commission run.

Interaction with commission

Commission restores components to their declared/last-known state. Without coordination, running commission after a manual toggle would silently undo the feature flag. Two mechanisms address this:

Scope guard: commission only reconciles components that are part of the solution being deployed — it never touches components outside the current solution. If the toggled flow is in the solution, commission will restore it; if it isn't (e.g. a flow from an earlier deploy or a different solution), commission ignores it.

--except flag: flowline commission prod --except "New Checkout Flow" skips named components during reconciliation. Keeps the toggle value intact through the next deploy.

Basis

direct: All SDK primitives are already used or established in the codebase — SetStateRequest in OrphanCleanupService.cs, workflow.statecode PATCH planned for Idea #1. Name resolution requires one query per component type; step names are already known to Flowline from the assembly registration pipeline. reasoned: The feature flag pattern is standard in software but has no Dataverse CLI equivalent. The community workaround is either direct Dataverse API scripts or navigating the maker portal per-component. A named toggle command covers the full activation lifecycle without a re-deploy.

Downsides

Name resolution can fail or be ambiguous — two flows with similar names, a step name that changed since last push. Should require exact match and fail clearly rather than fuzzy-matching. The toggle command operates outside Flowline's normal "source of truth" model — it changes runtime state not reflected in any source file, which is intentional but means drift detection (if ever built) needs to account for it.

Is Secure Config Still Worth It? Env Vars vs Secure Config

Dataverse environment variables have matured significantly. PAC handles them at import time via a settings file (pac solution create-settings), they're solution-aware, and they're accessible both in Power Automate flows natively and in plugins via an SDK query against environmentvariablevalue. The obvious question: does Flowline even need to implement secure config injection (Idea #4) now that env vars exist?

Environment Variables

  • ✓ Solution-managed — exported and versioned with the solution
  • ✓ PAC handles value configuration at import time (--settings-file)
  • ✓ Accessible in flows natively — no SDK query
  • ✓ Accessible in plugins via environmentvariablevalue query
  • ✓ One mental model for the whole team
  • ✗ Values visible to anyone with read on environmentvariablevalue — not a secret store
  • ✗ Not encrypted at rest in any special way
  • ✗ No access control per variable — it's a table, not a vault

Secure Config

  • ✓ Encrypted at rest in Dataverse
  • ✓ Admin-only readable — not visible to solution editors or service accounts
  • ✓ Per-step granularity — different secret per step, no cross-contamination
  • ✓ Plugin constructor receives it directly — no SDK query needed at runtime
  • ✗ Not exported in solutions — Flowline (or someone) must set it per deployment
  • ✗ No PAC CLI support — must be set via the SDK or admin console
  • ✗ Admin-only to read means you cannot verify it programmatically without elevated access
  • · No value type enforcement — plain string, however structured

The key differentiator is access control. In a Dataverse environment with multiple admins, consultants, service accounts, and maker roles, environmentvariablevalue is readable by a wide audience — the standard Dataverse security model does not restrict it to admins. If you store a Stripe API key, a webhook signing secret, or an OAuth client secret in an env var, any system admin or consultant onboarded to that environment can read it. That may be acceptable in many organizations; it is not acceptable in others.

Secure config addresses this specifically: the sdkmessageprocessingstepsecureconfig table is admin-only by platform design, the value is encrypted, and it never leaves Dataverse. A service account running flows cannot read another plugin step's secure config.

Verdict Env vars replace secure config for ~80% of use cases — non-sensitive config values, feature flags, tenant IDs, URLs, template strings. For the remaining 20% — webhook signing secrets, payment gateway keys, OAuth client credentials, anything that gives access to a third-party system — secure config still has a meaningful access-control advantage that env vars cannot match without a separate vault integration.

Recommendation: if your plugins don't use secrets (just config), skip Idea #4. If any plugin step calls an external API with a credential that should not be visible to all admins, Idea #4 is worth implementing — it's the only deployment-time mechanism that keeps that credential encrypted and admin-gated inside Dataverse. Env vars are not a drop-in replacement for secrets.

Recommended Order — Implementation Waves

All 7 ideas are independent enough to build in any order, but some decisions gate others and some make future work cheaper. Two anchor rules: #6 (IPostDeployService) should be introduced alongside the first post-deploy service, not in isolation; and #7 (Commission architecture) should be decided before building Wave 2 — it determines where flow activation and secure config injection live.
