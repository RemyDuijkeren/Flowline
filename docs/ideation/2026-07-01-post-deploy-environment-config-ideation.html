Classic Workflow State Restoration

Flow & workflow lifecycle Conf 90% Complexity: Low

Description

Solution import silently resets classic workflows and BPFs to Draft state — every deployment to any non-dev environment currently breaks them without error. Flowline takes a pre-import snapshot of workflow.statecode for all components in the solution, then post-import restores them via SetStateRequest. The developer never manually re-activates workflows again. A --no-restore flag provides an opt-out.

Basis

direct: Brainstorm 2026-06-12-deploy-state-restoration-requirements.md covers exactly this scenario (status: idea). OrphanCleanupService.cs already calls UpdateAsync with statecode/statuscode for workflow deactivation before deletion — the SDK pattern is proven in the codebase. Post-import hook at DeployCommand.cs:67 is the insertion point.

Rationale

The pattern (pre-import snapshot → post-import restore) is structurally identical to how OrphanCleanupService works. This is the second service following the same shape — no new architecture. The existing brainstorm already scoped it; this validates it as the right next implementation candidate.

Downsides

Restoring a workflow that was deactivated intentionally in the target environment — by a support engineer or admin outside of Flowline — would silently undo that. The --no-restore flag handles opt-out, but drift detection (Idea 6) is the fuller answer.

Per-Environment Component State in Configure File

Plugin step state & config Conf 85% Complexity: Low

Description

Plugin steps, flows, and workflows are always enabled by default after deploy — you wrote the code because you want it running. There is no IsDisabled attribute in StepAttribute and there shouldn't be: disabling a step globally via code makes no sense when the use case is always per-environment. Instead, a per-env configure file declares which components should be inactive in that specific environment. flowline configure uat reads the file and applies the overrides.

This enables the feature-flag pattern at the declarative level: a new flow ships dark in UAT because the configure file says "Active": false, while the same flow is live in prod the moment the prod configure file is updated. No code change, no re-deploy.

Configure file — state overrides section

// uat.configure.json — checked into source control (no secrets)
{
  "EnvironmentVariables": [ /* ... */ ],
  "ConnectionReferences": [ /* ... */ ],
  "Flows": [
    { "Name": "New Checkout Flow", "Active": false },
    { "Name": "Order Processing", "Active": true }
  ],
  "PluginSteps": [
    { "Name": "MyNs.BetaPlugin: Create of account", "Active": false }
  ]
}

Relation to inline mode (Idea #8)

Idea #8 extends flowline configure with inline mode (flowline configure prod flow "Name" --on) and interactive picking (--pick). This idea (file mode) is for repeatable environment setup: declared in the config file, committed to source control, applied on every deploy. If someone does an inline configure in UAT, the next file-mode run resets it to the declared state — the file is authoritative for environment setup. Inline mode is for urgent one-offs and release gates; file mode is for permanent declared state.

Basis

reasoned: Disabling a step via a code attribute ([Step(IsDisabled=true)]) would make it globally disabled in every environment — the opposite of what per-env feature flagging needs. The distinction is: code attributes describe the component's intent; environment config describes the environment's expectations. These are orthogonal concerns and must be in separate files. The configure file pattern (Idea #7) is already proposed; this idea defines what its state-override section looks like.

Downsides

Requires maintaining per-env configure files (one per environment: uat.configure.json, prod.configure.json). Only the components explicitly listed are overridden — everything else deploys active. Components removed from the solution are not covered (configure doesn't clean up deleted components' state). Naming the flow/step correctly in the file requires knowing the exact registered name.

Pre-Flight Config Validation

Settings & env vars Conf 85% Complexity: Low

Description

Before issuing pac solution import, Flowline validates that all environment variables and connection references required by the solution exist and are configured in the target environment. If anything is missing, deploy aborts with a clear actionable list: "Deploy blocked: 2 env vars missing values, 1 connection reference unconfigured." No partial import, no silent runtime failure on the first user action in prod.

Basis

direct: DeployCommand.cs already runs ValidateTargetAsync and ValidateDtapGateAsync before import — a validation checkpoint exists. SolutionChangeSummary.cs:334-338 already parses env var definition files from solution XML, establishing the file-reading capability. Solution XML in Package/src/ contains connection reference and env var definitions — readable before import.

Rationale

A failed import wastes 2–10 minutes and can leave the environment in a partially-updated state. Pre-flight makes deploy deterministic: it either succeeds completely or fails cleanly before touching the environment. This is the safety net that makes every other config feature trustworthy.

Downsides

Requires querying the target environment before import — adds a round-trip. --force flag should bypass as an escape hatch when the environment is intentionally partially configured.

Secure Config Deploy-Time Injection

Plugin step state & config Conf 80% Complexity: Medium

Description

Flowline reads a per-environment gitignored file — e.g. .flowline/secure-config.prod.json — and upserts sdkmessageprocessingstepsecureconfig records via SDK post-import. Webhook secrets, API keys, and signing certs get applied automatically as part of flowline deploy prod. No admin console, no manual post-deploy ritual, no bespoke per-project C# script.

The gitignored file pattern keeps secrets out of source control (honoring Flowline's design stance) while making the deployment reproducible. The format mirrors the step name already known to Flowline from registration: { "MyNamespace.StepName: Create of account": "secret-value" }.

Basis

direct: sdkmessageprocessingstepsecureconfig is a separate Dataverse table, not exported in solutions, admin-only read, must be set via API per deployment — confirmed by community research and Microsoft Learn. Current community workaround is bespoke C# scripts per project. external: The ALM Accelerator deployment settings payload (deprecated, no new features) is the most complete prior art. It defines two pipeline-variable keys per step:

# ALM Accelerator UserSettings payload keys:
sdkstep.unsec.<sdkstepid-guid> → unsecure (plain) config for the step
sdkstep.sec.<sdkstepid-guid> → secure (encrypted) config for the step

How it works: the export pipeline generates a customDeploymentSettings.json with token placeholders (#{sdkstep.sec.00000000-…}#). At deploy time, Azure DevOps' Replace Tokens task substitutes each placeholder with a matching pipeline variable (stored as a secret in Azure DevOps). The pipeline then writes the resolved value to the sdkmessageprocessingstepsecureconfig table via the Dataverse API. The step is keyed by its raw GUID — no human-readable name. Full two-file format in the deployment configuration guide.

How Flowline differs: instead of step GUIDs (opaque, change between environments), Flowline uses the human-readable step name already known from assembly registration (e.g. "MyNs.MyPlugin: Create of account"). Instead of Azure DevOps token replacement, Flowline uses a gitignored per-env JSON file — works in any CI system and locally without extra pipeline tooling.

Rationale

Secure config is the most cited reason Power Platform solutions "deploy fine, break in prod." Every plugin step that uses secure config is a silent manual dependency on every environment setup. The gitignored file pattern is CI/CD-native — secrets live in CI environment variables or a vault, the per-env file is generated at deploy time, never committed.

Downsides

Flowline handles secrets during the deploy step — requires careful handling and clear documentation that the file must be gitignored. Existing Flowline design stance explicitly says secure config "is intentionally not supported" (in reference to storing it in the unsecure attribute). This idea works differently (API call, not in source), but the stance may need to be updated.

Design stance note The prohibition in StepAttribute.Config is about unsecure config carrying secrets visible in solution XML — not about calling the secure config API post-deploy. This idea doesn't violate that stance, but the docs should be updated to clarify the distinction.

Ordered Flow Activation

Flow & workflow lifecycle Conf 75% Complexity: Medium

Description

All cloud flows deactivate on import. Activating them in arbitrary order causes silent failures when a parent flow calls a child that's still in Draft state. Flowline activates flows in dependency order — either inferred from flow dependency graphs in the solution XML (clientdata field, implementation risk flagged) or declared explicitly in a config block. Either approach calls workflow.statecode PATCH post-import, activating leaf flows first.

The declared config approach avoids the XML-parsing risk and aligns better with Flowline's convention-over-config philosophy — a short flowline.flows section in .flowline lists activation order and ownership per environment.

Basis

external: ALM Accelerator (deprecated 2023) documented activateflow.activate + activateflow.order + activateflow.activateas for exactly this. Community pain is well-evidenced: "connection reference was configured, flow still off after import" is a repeated forum pattern. The problem is confirmed by the flow-dependency ordering failure mode. direct: DataverseContextGenerator.cs:565-596 already reads connection references from solution XML — the XML-reading infrastructure exists.

Rationale

The first deploy to UAT or prod after adding a new flow often fails on the first customer trigger — not during deploy. Debug time is 20–40 minutes just diagnosing "child flow is Draft." Ordered activation eliminates this class of silent failures. Flow ownership reset (flows running as the wrong user post-import) is a natural companion feature.

Downsides

The clientdata JSON flow-dependency parsing approach is unverified — it's a promising implementation path but requires a real solution XML sample to confirm reliability. The declared config approach is safer but requires user-maintained ordering config. Start with declared, add inferred as a follow-on.

Implementation risk The clientdata-based inferred dependency graph is unverified against real solution XML. Recommend starting with declared ordering in config and treating inferred as a Phase 2 capability once parsing is confirmed.

IPostDeployService Protocol

Integration model Conf 85% Complexity: Medium

Description

Formalize OrphanCleanupService's ad-hoc pre/post import hooks into a typed IPostDeployService interface registered via DI. Each new capability (workflow restoration, step state, flow activation, secure config) becomes a service implementing that interface — the deploy command never changes. Each service is independently testable and shippable.

Basis

direct: DeployCommand.cs already calls orphanCleanupService.RunPreImportAsync() and RunPostImportAsync() — the pattern exists as ad-hoc calls. Formalizing it as a typed interface is the meta-investment that prevents each new capability from branching deploy command logic. The grounding notes "adding env config adds another chain link" — this makes the chain itself a protocol.

Rationale

Without this, every feature (workflow restore, step state, flow activation, secure config) modifies DeployCommand and creates N merge points. With it, DeployCommand stays stable, each capability is independently tested and shippable, and the order of execution is explicit. This is the leverage investment that makes ideas 1, 2, 4, and 5 all cheaper to build.

Downsides

Small refactor required to pull OrphanCleanupService's existing calls behind the interface. Adds indirection — must be worth it when more than 2 services exist. Build this alongside the second service implementation, not in isolation.

flowline configure — Post-Deploy Reconciliation

Integration model Conf 82% Complexity: Medium

Description

flowline deploy installs: pre-flight validation, pack, import, orphan cleanup. A separate flowline configure command applies the per-environment configure file: restoring workflow state, activating flows in dependency order, applying component state overrides, injecting secure config. The name is intentional — you hand it a settings artifact and it reconciles the environment to match. No magic, no inference.

flowline configure is idempotent and re-runnable at any time. After a support engineer manually disables a flow, re-running flowline configure prod restores declared state without a re-deploy.

The configure file — extended settings format

Reuses and extends the PAC solution settings file format. The base sections (EnvironmentVariables, ConnectionReferences) are identical to what pac solution create-settings generates — no new format to learn. Flowline adds sections for flow/workflow/step state and secure config. One file per environment, checked into source control (secure config entries go in a separate gitignored companion file).

// prod.configure.json — checked into source control
{
  // PAC-native sections (passed through to pac solution import)
  "EnvironmentVariables": [
    { "SchemaName": "cr123_FeatureFlag", "Value": "true" }
  ],
  "ConnectionReferences": [
    { "LogicalName": "cr123_SharedO365", "ConnectionId": "abc123" }
  ],
  // Flowline-extended sections (applied post-import)
  "Flows": [
    { "Name": "Order Processing", "Active": true },
    { "Name": "Beta Checkout Flow", "Active": false }
  ],
  "Workflows": [
    { "Name": "Approval Process", "Active": true }
  ],
  "PluginSteps": [
    { "Name": "MyNs.BetaPlugin: Create of account", "Active": false }
  ]
  // SecureConfig goes in gitignored prod.configure.secrets.json
}

Ownership split

Deploy owns

  • Pre-flight validation
  • Pack solution
  • Import solution (PAC CLI)
  • Orphan cleanup

Configure owns

  • Classic workflow state restoration
  • Cloud flow activation (ordered)
  • Per-env component state overrides
  • Secure config injection

Default behaviour & flags

# Default: configure runs automatically after deploy
flowline deploy prod

# Deploy only — skip configure (CI step 1)
flowline deploy prod --no-configure

# Configure only — re-apply without re-deploying
flowline configure prod

# Dry-run: show what would change without applying
flowline configure prod --dry-run

# Exclude components (preserve intentional toggle state)
flowline configure prod --except "Beta Checkout Flow"

Why split matters

Deploy runs on every merged PR — fast, automated, no secrets required. Configure runs after, either automatically or gated by a human go/no-go. Secrets for secure config only need to be present in the configure step. In regulated environments these can have different approvers: DevOps runs deploy; solution architect runs configure to confirm activation is intentional.

Basis

direct: The brainstorm 2026-06-12-environment-config-command-requirements.md proposes a flowline config command — configure is that direction with a clearer name. Reusing the PAC settings file format keeps the mental model tight. reasoned: Configure implies "I hand you a file and you apply it" — this matches what the command does. It's analogous to Terraform apply: a declared state file drives reconciliation.

Downsides

flowline configure could be confused with configuring Flowline itself. Consider flowline apply as an alternative if UX research shows that friction. Maintaining per-env configure files alongside deploy settings files is two files per environment — mitigated if flowline create-configure prod generates the template.

flowline configure — inline & interactive mode

Operations Integration Conf 87% Complexity: Low

Description

flowline configure (Idea #7) handles environment setup from a config file. The same command should also work inline — no file needed — for one-off changes, feature-flag gates, and instant rollbacks. Two syntaxes, same command, same SDK calls:

  • File mode: flowline configure prod or flowline configure prod --settings-file deploymentSettings.prod.json — reads a config file and reconciles the full environment.
  • Inline mode: flowline configure prod flow "My Flow" --on — targets one component directly on the command line. No file needed.
  • Interactive mode: flowline configure prod --pick — queries the live environment and presents a selection list; user picks without knowing the name.

This is the feature-flag primitive: ship a flow as Draft, enable it when the go-live decision is made, roll back instantly if something breaks — all without re-deploying the solution.

Command surface

# File mode — apply full settings file (Idea #7 behavior)
flowline configure prod
flowline configure prod --settings-file deploymentSettings.prod.json

# Inline mode — target one component directly, no file needed
flowline configure prod flow "New Checkout Flow" --on
flowline configure prod flow "New Checkout Flow" --off
flowline configure uat workflow "Order Approval" --on
flowline configure prod step "MyNs.NewFeaturePlugin: Create of account" --off
flowline configure prod envvar cr123_EnableNewCheckout true

# Dry-run works on both modes
flowline configure prod flow "New Checkout Flow" --on --dry-run

# Interactive mode — browse + pick from live environment (no name needed)
flowline configure prod --pick
flowline configure prod flow --pick

What each operation calls under the hood

Cloud flow / classic workflow: workflow.statecode PATCH — 1 = Activated, 0 = Draft. Inline and file mode both call the same SDK primitive.

Plugin step: SetStateRequest on sdkmessageprocessingstep — State 0 = Active, State 1 = Inactive.

Env var: environmentvariablevalue upsert — create or update the per-environment value record.

Secure config: sdkmessageprocessingstepsecureconfig upsert — same mechanism as Idea #4, available both inline and via the configure file.

File mode vs inline mode

Both modes call the same SDK primitives. The difference is the source:

File mode (flowline configure prod or --settings-file) — reads a config file, reconciles all declared components. Git-tracked, reproducible, CI-safe. The authoritative state for the environment. Declare component state in the file with an Enabled key:

"Flows": [
  { "Name": "New Checkout Flow", "Enabled": false },  // ships dark
  { "Name": "Legacy Order Flow", "Enabled": true }
]

Inline mode (flowline configure prod flow "Name" --on) — targets one component directly, no file needed. For ad-hoc changes, urgent rollbacks, or scripted release gates that aren't worth a config file edit. After running, Flowline can emit the equivalent config file snippet so the decision can be committed if it should be permanent.

Scope guard: file mode only reconciles components listed in the file. A component toggled inline that isn't in the file is left untouched by the next file-mode run. Use --except "Name" to skip a named component even if it appears in the file.

Interactive picker mode

Knowing the exact component name is a barrier — especially for env vars (schema names) and plugin steps (fully qualified names). An interactive mode queries the live environment and presents a selection list so the user browses and picks without needing to look anything up.

# Browse all component types in the environment
flowline configure prod --pick

# Narrow to one component type before picking
flowline configure uat flow --pick
flowline configure prod step --pick

The picker queries Dataverse, groups results by type (Flows, Workflows, Plugin Steps, Env Vars), and shows current state (Active / Draft / value) alongside each name. The user multi-selects, chooses enable or disable, confirms, and Flowline applies. Spectre.Console's MultiSelectionPrompt makes this a ~30-line implementation — Flowline already depends on Spectre.Console for all CLI rendering. After applying, Flowline can emit the equivalent inline commands and the config file snippet so the session is reproducible.

Limitation: requires a live connection to the target environment. Cannot run offline or in a headless CI pipeline without --env resolving to a reachable environment. For CI use, the configure file (declarative) is the right surface — interactive picker is a local developer tool.

Basis

direct: All SDK primitives already exist in the codebase — SetStateRequest in OrphanCleanupService.cs, workflow.statecode PATCH planned for Idea #1. flowline configure already exists (Idea #7) — inline mode is an additive parser path, not a new command. Spectre.Console MultiSelectionPrompt is available in the existing Spectre.Console dependency. reasoned: The feature flag pattern is standard in software but has no Dataverse CLI equivalent. The community workaround is direct API scripts or navigating the maker portal per-component. Folding inline mode into flowline configure gives users one command to learn instead of two. The interactive picker removes the name-lookup friction that makes ad-hoc changes impractical today.

Downsides

Name resolution can fail or be ambiguous — two flows with similar names, a step name that changed since last push. Non-interactive mode should require exact match and fail clearly rather than fuzzy-matching. The toggle command operates outside Flowline's normal "source of truth" model — it changes runtime state not reflected in any source file, which is intentional but means drift detection (if ever built) needs to account for it. Interactive picker has latency (Dataverse query per component type) — fine for developer use, not CI.

PAC Solution Settings File at Import

Settings & Env Vars Conf 79% Complexity: Low

Description

The PAC CLI natively supports a --settings-file flag on pac solution import. The settings file (generated by pac solution create-settings) maps environment variable schema names and connection reference logical names to per-environment values. Flowline currently does NOT pass this flag — after deploy, users must manually configure env vars and connection refs in the maker portal, or use a separate PAC command. This idea wires up the passthrough: Flowline discovers the settings file for the target environment and forwards it to PAC during import.

The settings file is the PAC-managed base of what Idea #7's configure file extends. Getting native PAC passthrough working first is the natural foundation: it handles env vars and connection refs at import time, which is when PAC processes them correctly.

Settings file format (PAC-native)

# Generate template (run once per solution) — outputs deploymentSettings.json
pac solution create-settings --solution-zip MySolution_managed.zip --settings-file deploymentSettings.json

// deploymentSettings.prod.json (rename + fill in values per environment)
{
  "EnvironmentVariables": [
    { "SchemaName": "cr123_ApiBaseUrl", "Value": "https://api.prod.contoso.com" }
  ],
  "ConnectionReferences": [
    { "LogicalName": "cr123_SharedO365", "ConnectionId": "abc123", "ConnectorId": "/providers/Microsoft.PowerApps/apis/shared_office365" }
  ],
  "CopilotAgents": [ ← confirmed 3rd section in PAC CLI output
    { "AadGroupId": "00000000-0000-0000-0000-000000000000", "Name": "msdyn_salesCopilot" }
  ]
}

# Flowline passes the matching per-env file automatically:
pac solution import --path MySolution.zip --settings-file deploymentSettings.prod.json

How Flowline discovers the settings file

Convention: Flowline looks for deploymentSettings.<env>.json in the solution's Package/ folder (where PAC already places deploymentSettings.json). If found, it passes it to PAC. If not found, deploy proceeds without it. No config required — convention over config. Workflow: generate with PAC → rename to deploymentSettings.dev.json / deploymentSettings.prod.json → fill in per-env values → commit.

The configure file from Idea #7 extends this: Flowline reads deploymentSettings.<env>.json, splits PAC-native sections (EnvironmentVariables, ConnectionReferences, CopilotAgents) → pac solution import --settings-file; Flowline-specific sections (Flows, PluginSteps, SecureConfig) → post-import reconciliation. One file, two consumers.

Is the PAC settings file widely used or a dead feature?

The PAC settings file exists since 2021 and is the official Microsoft-recommended approach for environment-specific configuration at import time. In practice, adoption varies: teams using ALM Accelerator or automated pipelines typically use it; manual-deploy teams often don't know it exists. The feature is not deprecated — it is actively maintained and appears in PAC CLI release notes. Community sentiment on the Power Platform forums suggests it is underused rather than avoided, primarily because discoverability is poor and the generate-then-edit workflow adds friction.

The main reason teams skip it: pac solution create-settings generates the template file, but you still need to fill in the values and keep the file in sync as the solution evolves. Without tooling to detect drift (new env var added to solution but missing from settings file), it silently fails to set values — the variable is left at its default. Idea #3 (pre-flight config validation) would address exactly this gap.

Pros

✓ Zero net code for env var and connection ref application — PAC already handles it
✓ PAC applies env vars during import, before flows are activated — correct ordering for free
✓ Settings file is solution-aware — PAC validates schema names against the solution being imported
✓ Convention file name (deploymentSettings.prod.json) matches PAC CLI output — familiar to teams already using PAC directly
✓ Plugs the current Flowline gap: deploy completes but env vars are unset until manual intervention

Cons

✗ Settings file must be manually maintained and committed to source control — values can drift from reality
✗ Connection IDs are env-specific GUIDs that change when a connection is recreated — file needs updating
✗ PAC validates schema names at import time; missing entry = silent skip (no value set, no error)
✗ Duplicate maintenance if teams use both PAC CLI directly and Flowline

Basis

direct: pac solution import --settings-file is a first-class PAC CLI feature documented at learn.microsoft.com. The settings file format is stable. Flowline already shells out to PAC for import — this is a one-flag addition to an existing call site. reasoned: Not wiring this up means every Flowline deploy leaves environment variables unset until a human follows up. That contradicts Flowline's promise of a complete, repeatable deploy.

Is Secure Config Still Worth It? Env Vars vs Secure Config

Dataverse environment variables have matured significantly. PAC handles them at import time via a settings file (pac solution create-settings), they're solution-aware, and they're accessible both in Power Automate flows natively and in plugins via an SDK query against environmentvariablevalue. The obvious question: does Flowline even need to implement secure config injection (Idea #4) now that env vars exist?

Environment Variables

  • ✓ Solution-managed — exported and versioned with the solution
  • ✓ PAC handles value configuration at import time (--settings-file)
  • ✓ Accessible in flows natively — no SDK query
  • ✓ Secret type env vars can reference Azure Key Vault — platform fetches value transparently (flows & connectors only — not available in plugins)
  • ✓ One mental model for the whole team (flows + plugins + Power Pages all read the same table)
  • ✗ Values visible to anyone with read on environmentvariablevalue — standard env vars are not a secret store
  • ✗ Plugin retrieval requires a manual SDK query against environmentvariablevalue — no platform helper; you write it yourself (typically in a base class)
  • ✗ Extra synchronous query inside plugin execution — adds latency relative to constructor injection
  • ✗ Key Vault-backed secret env vars do NOT work in plugins — Microsoft explicitly limits them to Power Automate flows, Copilot Studio, and custom connectors; a plugin query against environmentvariablevalue returns no value for KV-backed secrets
  • ✗ Key Vault-backed env vars require Azure managed identity configuration — non-trivial operational overhead

Secure Config

  • ✓ Encrypted at rest in Dataverse
  • ✓ Admin-only readable — not visible to solution editors, service accounts, or makers
  • ✓ Per-step granularity — different secret per step, no cross-contamination
  • ✓ Passed to plugin constructor directly — zero extra SDK queries at runtime, fastest possible access pattern
  • ✓ Still actively used in 2024–2026 — particularly in ISV and regulated-sector projects where admin read access must be tightly controlled
  • ✗ Not exported in solutions — must be set per deployment via SDK or admin console (no PAC CLI support)
  • ✗ Plugin-only — flows, canvas apps, and Power Pages cannot read secure config at all
  • ✗ Admin-only read means you cannot verify or diff the current value programmatically without elevated access
  • · No value type enforcement — plain string, however structured (often JSON)

Can env vars connect to Azure Key Vault?

Yes — Dataverse has a Secret type for environment variables (GA since 2023). A Secret env var stores a reference to an Azure Key Vault secret rather than a value directly; the platform retrieves the current secret version at runtime. Critical limitation: KV-backed secret env vars are explicitly limited to Power Automate flows, Copilot Studio agents, and custom connectors. Plugins cannot retrieve the secret value — Microsoft documentation states the secrets "aren't available for use in other customizations or generally via the API." A plugin querying environmentvariablevalue for a KV-backed secret receives no value.

Practical implication for plugin developers: Azure Key Vault is not an option for plugin secrets, regardless of infrastructure investment. Secure config remains the only mechanism for delivering encrypted, admin-controlled credentials directly to a plugin constructor — without a runtime query and without an SDK workaround. Key Vault-backed env vars solve the same problem in flows and connectors, but not in plugins.

Operational note: KV-backed env vars require an Azure managed identity or app registration in the Dataverse environment, plus Key Vault access policy configuration. Used by larger enterprise teams with existing AKV infrastructure. For teams without AKV, secure config is simpler for the plugin credential use case.

Plugin retrieval: query vs constructor injection

This is the clearest practical difference for plugin developers. Env var retrieval requires an SDK query every time a plugin needs the value:

// Env var — you write this query (typically in your base class)
var q = new QueryExpression("environmentvariablevalue");
q.Criteria.AddCondition("environmentvariabledefinitionid", ConditionOperator.Equal, defId);
var val = svc.RetrieveMultiple(q).Entities.FirstOrDefault()?.GetAttributeValue<string>("value");

// Secure config — platform does it for you
public MyPlugin(string unsecureConfig, string secureConfig) {
  _apiKey = secureConfig; // already there, no query
}

Secure config is faster inside plugin execution because the value arrives with the execution context — no extra round-trip to Dataverse. For plugins that fire on high-volume tables (account updates, email sends), this is a real performance difference. Env var queries are cached by the platform after the first call within an execution, but the first call adds latency.

Community usage in 2024–2026

Secure config is not dead. It remains the standard pattern for plugin steps that call external APIs with credentials (payment gateways, email providers, webhook signing keys). The Power Platform community has not abandoned it — it is still recommended in official Microsoft guidance for plugin secrets. What has changed: for non-secret config (URLs, feature flags, tenant IDs, template strings), env vars are now the default recommendation and secure config is not mentioned. The split is roughly: config = env vars, credentials = secure config.

ISV packages targeting AppSource frequently use secure config because it prevents customers from reading the ISV's licensing key or API credentials, even as tenant admins. This use case has no env var equivalent without AKV, which many customer environments don't have configured.

The key differentiator is access control and retrieval model. In a Dataverse environment with multiple admins, consultants, service accounts, and maker roles, environmentvariablevalue is readable by a wide audience — the standard Dataverse security model does not restrict it to admins. If you store a Stripe API key, a webhook signing secret, or an OAuth client secret in a standard env var, any system admin or consultant onboarded to that environment can read it. AKV-backed Secret env vars solve this at the cost of AKV infrastructure.

Verdict Env vars replace secure config for ~80% of use cases — non-sensitive config values, feature flags, tenant IDs, URLs, template strings. For the remaining 20% — webhook signing secrets, payment gateway keys, OAuth client credentials — secure config still has a meaningful advantage: zero-query constructor injection (faster), admin-only read (tighter access control), no AKV infrastructure required (simpler).

Key Vault-backed env vars close the secret-storage gap but add operational complexity. They're the right choice when you already have AKV in the enterprise stack; secure config is simpler when you don't.

Recommendation for Flowline: implement Idea #4 (secure config injection). It handles the credential use case that env vars — even AKV-backed ones — don't cover without extra infrastructure. The plugin constructor injection pattern is objectively simpler and faster than the env var query pattern for secrets that only plugins need.
