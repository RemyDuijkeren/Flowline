Codebase Context

Grounding Context

What Spectre.Console does today

  • 9 independent AnsiConsole.Progress() blocks (6 plugin + 3 web resource phases) — cannot nest, each vanishes on completion
  • ProgressTask? threaded as optional parameter through PluginExecutor / WebResourceExecutor
  • Console.Status() spinners for validation checks
  • Custom Ok() / Done() / Skip() / Warning() / Error() extension methods
  • IRenderHook (LoggingRenderHook) auto-captures all Spectre output to Serilog — no per-call plumbing
  • No AnsiConsole.Live(), no Layout, no persistent state display

Deploy pipeline shape

  • Sequential validation → parallel stage ops → post-deploy fan-out
  • Plugin ops: leveled-sequential (types → steps → images)
  • Web resource ops: bounded parallel (MaxParallelism=8, thread-safe lock(failures))
  • Failures collected in List<(string Name, Exception Error)>, reported at batch end
  • FlowlineException with ExitCode enum (13 codes, incl. PartialSuccess=18)
  • Cancellation token wired end-to-end
  • SolutionChangeSummary.WriteTree() for hierarchical component rendering

CI and interactive detection

  • ConsoleHelper checks CI, TF_BUILD, JENKINS_URL, GITHUB_ACTIONS
  • AnsiConsole.Profile.Capabilities.Interactive for TTY check
  • --force auto-accepts confirmations in CI
  • Existing flags: -v/--verbose, -f/--force, --no-cache
  • No --no-progress, --plain, or --quiet flag
  • SelectionPrompt for profile disambiguation; TextPrompt.Secret for SP auth

Spectre.Console capabilities available

  • AnsiConsole.Live(renderable) — updates any IRenderable in-place
  • Layout with SplitColumns() / SplitRows() for multi-panel views
  • Table, Tree, Panel as IRenderable targets
  • SelectionPrompt / MultiSelectionPrompt for pre-run selection
  • Raw Console.KeyAvailable polling inside Live callback (workaround for #1452)
Two hard constraints shape every idea here: (1) Dataverse asyncoperation.percentcomplete returns HTTP 400 — all Dataverse ops are binary (In Progress → Succeeded). No percentage bars, ever. (2) Spectre.Console SelectionPrompt / interactive prompts cannot coexist with AnsiConsole.Live() (issue #1452, "not supported by design"). The workaround: front-load all prompts before Live() starts, or use raw Console.KeyAvailable polling inside the Live callback.

1.

CI-Native Output Mode

ci-compatibility Conf 85% Cx Low-Med
Description

Auto-detect CI environment variables — CI, TF_BUILD, JENKINS_URL, GITHUB_ACTIONS — and switch to platform-native log grouping instead of ANSI progress bars. On GitHub Actions, emit ::group::plugin-registration / ::endgroup:: at each phase boundary. On Azure Pipelines, emit ##[section] / ##[endsection]. Each of the 9 deployment phases becomes a collapsible section in the CI log UI automatically. Add a --json-events flag that routes phase-transition events as newline-delimited JSON to stdout ({"phase":"plugins","status":"done","elapsed_ms":2100}) while ANSI renders to stderr. No --no-progress flag needed — CI is already detected.

The detection logic is half-built; the output routing is the missing half.

Basis

direct: CI detection already in ConsoleHelper.cs:33-36 (CI, TF_BUILD, JENKINS_URL, GITHUB_ACTIONS) — detection is confirmed
direct: No --no-progress, --plain, or --quiet flag exists — confirmed gap
external: GitHub Actions workflow commands spec (::group:: / ::endgroup::); Azure Pipelines logging commands (##[section])
external: gh --json, aws --output json — major CLI tools treat structured output as a first-class mode

Rationale

CI is a confirmed gap — no structured or clean-text output path exists. Detection is already half-built. This doesn't require the Unified Phase Rail (A1): phase-boundary hooks (Console.WriteLine("::group::plugins") calls at phase start/end) are simple and independent. The Flight Data Recorder framing applies: same deploy events, two audiences — humans watch the ANSI cockpit display; CI systems read the structured recorder stream.

Downsides: Routing ANSI display to stderr requires AnsiConsoleSettings { Out = AnsiConsoleOutput.FromStream(Console.Error) } — supported by Spectre but a non-trivial configuration change to the console setup. The ::group:: approach helps GitHub Actions and Azure Pipelines but not every CI system. --json-events to stdout requires identifying all phase-boundary call sites.
2.

Unified Phase Rail (Central State Model)

live-progress Conf 70% Cx High
Description

Replace the 9 independent AnsiConsole.Progress() blocks — which cannot nest, each vanishing on completion — with a single AnsiConsole.Live(Layout) session owned at the orchestration layer. Services write phase status to a shared PipelineState record rather than owning their own Progress lifecycle. One central renderer reads the state and repaints on each tick: completed phases stay visible as ✓ rows, the active phase shows a spinner and elapsed time, upcoming phases show as pending.

This is the Toyota Andon Board applied to CLI deploys: all stations simultaneously visible in a persistent display, not sequentially flashing and vanishing. It's the keystone idea — Ideas 3, 4, and 5 become additive subscribers to the shared state model once this is in place.

Basis

direct: ProgressTask? threaded as optional parameter through PluginExecutor and WebResourceExecutor — executors were designed to receive context from outside, not own the Progress lifecycle
direct: 9 independent AnsiConsole.Progress() blocks confirmed — they cannot nest; no unified view is possible without this architectural change
reasoned: A shared PipelineState record makes every display mode (terminal, CI text, end-of-run summary) a subscriber; implementation cost is paid once, not once per display concern

Rationale

The root complaint is orientation: users deploying a 9-phase pipeline have no idea where they are. Each Progress block vanishes so there's no visible record when a late phase fails. The unified rail directly fixes this — "Phase 3 of 9: Uploading web resources" — and the persistent display becomes the terminal-as-receipt (Idea 4) when the run ends. Everything else that's interesting about the dashboard builds on top of this change.

Downsides: High implementation effort for a solo dev — executor method signatures may need changing to receive an external Live session handle. The event bus pattern (4 components: interface, state model, renderer, CI subscriber) is elegant but substantial. Can be shipped incrementally: a simple shared Table first, refactored to an event bus later. Implementation is the gate for Ideas 4 and 5.
3.

Deploy Preview Tree

live-progress Conf 90% Cx Low-Med
Description

Before any Dataverse mutation begins, scan the solution and render the full operation tree as a static display: plugin types with step counts, web resource files by type, solution version, target environment. The user sees "here is everything about to happen" in under 2 seconds. Then existing Progress blocks run the deploy as they do today.

This is a pre-flight manifest, not a live tracker — rows don't check off during execution (that requires the Unified Phase Rail). But it directly eliminates the 30-second blank-screen anxiety at source: if you can see the full operation list up front, a running spinner has context. It also catches misconfiguration before anything writes to Dataverse.

Basis

direct: Plugin attribute scanning (types → steps) already happens before any CRM API call — the count is a byproduct of scanning, not a new operation
direct: Web resource filenames are filesystem-local — count and list from WebResources/src/ before any upload starts
direct: SolutionChangeSummary.WriteTree() proves the hierarchical tree rendering pattern already exists in the codebase

Rationale

Reframes progress from reactive ("what just happened?") to declarative ("here's what's about to happen"). The deploy shape is fully knowable before any mutation — plugin steps from assembly attributes, web resources from the filesystem, solution metadata from .flowline config. No Dataverse API call needed. This idea stands on its own and doesn't require the Unified Phase Rail.

Downsides: The static tree doesn't update as execution proceeds — users see the plan but not live progress. May feel incomplete compared to the full Unified Phase Rail. A 2-second pre-scan adds latency before execution starts (likely imperceptible for normal solution sizes).
4.

Deploy Receipt (JSON file)

end-of-run-summary Conf 90% Cx Low
Description

After every completed deploy, write a structured receipt to .flowline/runs/<yyyyMMdd-HHmmss>.json — phases executed, per-phase timing, component counts (plugins registered, web resources synced, orphans cleaned), exit code, and any failure details. Terminal output is ephemeral: SSH disconnects, terminal closures, and CI log truncation all lose it. The receipt file is permanent.

A natural companion command flowline history — out of scope for this idea but an obvious next step — would render past runs as a table, answering "what did I deploy last Thursday and did it succeed?" without leaving the tool.

Basis

direct: FlowlineException with ExitCode enum (13 codes, incl. PartialSuccess=18) — structured end-state representation already exists; receipt captures it durably
direct: SolutionChangeSummary already builds structured component change data — the receipt schema is implicit in existing types
reasoned: Terminal output is ephemeral by design; a deploy to a production environment is a permanent event. Permanence of the event record should match permanence of the event itself.

Rationale

Every other summary idea requires the Unified Phase Rail (Idea 2). The receipt file does not — it's a post-run write operation that works with the current architecture. It also provides the historical timing data that Idea 5 (ICU Baseline) needs. Implementing Idea 4 first creates the infrastructure both Idea 5 and a future flowline history command can build on.

Downsides: .flowline/runs/ accumulates files and needs a retention policy (or manual cleanup). The JSON schema needs a versioning story as Flowline evolves. flowline history is a separate design and implementation surface not included here.
5.

ICU Timing Baseline

live-progress Conf 80% Cx Medium
Description

Store per-phase wall-clock durations in .flowline/history.json (or within the deploy receipt from Idea 4) after each run. During the next deploy, show each phase's current elapsed time vs. its P50 from the last N runs: Uploading web resources · 90s (usual: 45s) ⚠. A phase running significantly past its baseline gets an amber indicator. A phase finishing well under baseline gets a subtle confirmation.

An ICU monitor never shows "% of heartbeat remaining" — that's unknowable. Instead it shows current rate against rolling trend: HR 112bpm vs. baseline 78. The deviation IS the signal. Dataverse forces the same constraint: no percentcomplete, but elapsed time is always available. Historical timing is definitionally the only ETA signal available.

Basis

direct: Dataverse asyncoperation.percentcomplete returns HTTP 400 — all ops are binary. If you cannot show a percentage, your only non-fake progress signal is elapsed time relative to history
direct: ValidationCacheStore.cs proves the local JSON cache pattern exists in this codebase — the storage mechanism is established
reasoned: Historical timing is not an approximation — it's the correct response to the constraint. A tool that says "usually 45s; currently at 90s" is more honest than a fake percentage bar

Rationale

Turns the "is this frozen?" question from anxiety into data. The most annoying part of a deploy isn't that it takes time — it's not knowing if it's making progress or stuck. "Uploading: 90s (usual: 45s) ⚠" tells you to investigate; "Uploading: 45s (usual: 47s)" tells you to relax. This is more useful than a fake progress bar and more honest than a spinner with no context. Requires Idea 4 (receipt JSON) as its storage mechanism, and either the Unified Phase Rail or a per-phase timing hook to collect the data during the run.

Downsides: Meaningless for the first 6 deploys (no baseline yet — needs a "first run, no baseline" display state). Baselines vary across environments and solution sizes — a baseline from a 10-component solution may mislead on a 100-component solution. P50 of 7 runs needs tuning. Requires Idea 4 for storage and either Idea 2 (Unified Phase Rail) or a targeted timing hook to capture per-phase durations.
6.

Pre-Flight Manifest Gate

interactive-selection Conf 75% Cx Low
Description

Before AnsiConsole.Live() starts (or before the first Progress block), render a pre-flight manifest: target environment, solution name, component counts (X plugins, Y web resources), and any destructive-operation count. User presses Enter to proceed or Q to cancel. --yes / --force skips in CI.

This is the clean solution to the Spectre #1452 constraint: all interactive decisions fully resolve before Live() begins. The prompt isn't fighting with a Live session — it's sequenced before one starts. This is the Kitchen Expediter model: the expediter reads the full ticket before any dish is fired.

Note: This is a manifest-display-and-confirm only. Phase-selection (deselecting phases to skip) is out of scope — the 9 deploy phases aren't registered as a common IService collection, and phase dependencies (leveled-sequential plugin phases) make skip logic non-trivial.

Basis

direct: Spectre issue #1452 — sequencing is the documented clean workaround: prompt finishes, then Live() starts; no Console.KeyAvailable hacking needed
direct: SelectionPrompt already used for profile disambiguation — the pre-flight prompt pattern is established
direct: --force already auto-accepts confirmations in ConsoleHelper.cs:60-61 — CI bypass is half-built; needs a --yes alias for non-destructive confirmations

Rationale

Prevents the most painful class of Dataverse deploy mistake: deploying the wrong solution to production. A pre-flight manifest that says "Deploying MyApp v1.3.2 to PRODUCTION — 23 plugin steps, 14 web resources" requires explicit confirmation before anything mutates. Costs one keypress on deliberate deploys, costs zero in CI via --force. Also cleanly exploits the one Spectre workaround that doesn't require Console.KeyAvailable polling.

Downsides: Adds a confirmation step to every deploy — experienced users who always deploy to one environment may find it friction. Needs a --yes flag addition. Some decisions (e.g., "how many orphans will be cleaned") are only known after scanning, not before — the manifest may be incomplete for some counts. Phase-selection feature is explicitly deferred due to dependency graph complexity.
7.

Confidence-First Silent Mode

live-progress Conf 60% Cx Low-Med
Description

Flip the default output: flowline deploy runs with a single spinner + current phase name only. The 9 Progress blocks become opt-in via --progress. The end-of-run summary (success/fail, component counts) always prints regardless of mode.

The philosophy: every other idea here ADDS display. This one REMOVES it. The 9 sequential Progress animations produce noise in CI logs and visual overload in interactive sessions. A single spinner showing continuous motion plus the phase name (⣷ Uploading web resources) is sufficient proof of liveness. Detailed progress becomes something you enable when debugging, not something you endure on every run. go build and make are silent on success — "silence as confidence" is a long-established Unix convention.

Basis

reasoned: CI detection already makes output reactive to context — flipping the human default to clean-by-default applies the same principle to interactive sessions
external: go build, make — silent on success, verbose on VERBOSE=1; "silence as confidence" is a Unix convention with broad validation

Rationale

The specific pain ("30-second blank screen triggers anxiety") is better addressed by one continuous spinner than by 9 animated progress bars. Silence here means "one line of motion" not "nothing" — the spinner proves the process is alive while the phase name provides context. This is also the lowest-effort of all dashboard ideas: just don't open 9 Progress blocks.

Downsides: Real counterargument: A deploy tool that mutates remote Dataverse state may have different visibility requirements than a local build tool. "What's deploying right now?" is a safety question when mistakes are costly to reverse. Flipping the default is a breaking UX change with no migration story. The Unix analogy (go build) doesn't hold perfectly — local compilation is easily retried; a Dataverse deploy mid-flight is not. This is a genuine product philosophy debate rather than an obvious win, which is why confidence is 60%. Needs an explicit user decision before shipping.
