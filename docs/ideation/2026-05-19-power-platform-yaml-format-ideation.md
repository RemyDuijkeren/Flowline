# Ideation: Power Platform Solution YAML Format

## Grounding Context

**Question investigated:** Can Flowline use the Power Platform solution source control YAML format (https://learn.microsoft.com/en-us/power-platform/alm/solution-source-control-yaml-format)?

**Codebase context:**
- Flowline: C#/.NET CLI, sync-first ALM for unmanaged Power Platform solutions
- Current path: `pac solution clone` → `src/` with XML format; `pac solution sync` for incremental updates; `pac solution pack --folder src/` to pack
- Strategy: small teams/consultants, PROD-as-truth, no managed solutions

**Investigation:**
- Fetched official PAC CLI docs for `pac solution clone`, `pac solution sync`, `pac solution unpack`
- Inspected actual clone output at `E:\Code\TryOut\MyFlowTest\solutions\Cr07982`
- Attempted `--format yaml` and `--sourceControl` flags on PAC commands — both rejected with "unknown argument" errors

**Key finding — the YAML format is not producible via PAC CLI:**
- `pac solution clone` produces **XML format** (`src/Other/Solution.xml`, `src/Entities/*/Entity.xml`, etc.) — confirmed from actual output
- `pac solution sync` operates on the same XML clone structure
- `pac solution unpack` produces XML by default
- `--format yaml` does not exist on any PAC command
- `--sourceControl` does not exist on any PAC command
- The YAML format (`solution.yml`, `solutioncomponents.yml`, `rootcomponents.yml`, `missingdependencies.yml`) is **only written by native Dataverse Git integration** — the feature in Power Apps maker portal that connects a solution to an Azure DevOps repo
- `pac solution pack` can *read* YAML-format repos, but nothing in PAC CLI *writes* them

**What PAC CLI clone actually produces (XML format):**
```
solutions/<Name>/
├── <Name>.cdsproj
├── src/
    ├── Other/Solution.xml          ← XML format marker
    ├── Other/Customizations.xml
    ├── Entities/<Name>/Entity.xml
    ├── Workflows/*.json            ← Power Automate flows (JSON, not YAML)
    ├── Workflows/*.xaml            ← classic background workflows
    ├── bots/                       ← Copilot Studio bots
    ├── customapis/
    ├── PluginAssemblies/
    ├── SdkMessageProcessingSteps/
    └── WebResources/
```

Modern flows (Power Automate cloud flows) ARE accessible in XML format as `.json` files — they are not YAML-only. Canvas apps appear as `.msapp` binary blobs in XML format — not source-readable.

## Topic Axes

1. pac command path — switching from export+unpack to clone/sync as Flowline internals
2. Manifest parsing — reading solutioncomponents.yml, rootcomponents.yml, missingdependencies.yml offline
3. Diff quality & drift detection — component-type-aware diffs and offline drift
4. Component type coverage — canvas apps and modern flows only in YAML
5. Repo layout — YAML puts components at repo root vs solutions/<Name>/src/; multi-solution sharing

## Ranked Ideas

### 1. Component-Type-Aware Diff from XML Folder Structure

**Description:** Use the existing XML clone folder structure to label post-sync diffs by component type. The XML layout already uses typed folders — `Entities/`, `Workflows/`, `bots/`, `customapis/`, `PluginAssemblies/`, `WebResources/` — which are sufficient to classify changed files. `SolutionChangeSummary` currently diffs files generically; wrapping it with folder-based type classification produces "3 entities changed, 2 flows changed, 1 web resource changed" instead of "17 files changed."

**Axis:** Diff quality & drift detection

**Basis:** `direct:` Actual clone output at `E:\Code\TryOut\MyFlowTest\solutions\Cr07982\src\` confirms typed folder layout; `direct:` `SolutionChangeSummary.cs` already performs post-sync file diffing.

**Rationale:** No format switch required. The XML folder structure already encodes component type in the path. This is buildable today from the existing clone output.

**Downsides:** Folder name to component type mapping must be maintained manually as PAC adds new component types. Sub-folder nesting (e.g., `Entities/<Name>/FormXml/`) means a "changed entity" may be dozens of files — need to aggregate at the entity level, not file level.

**Confidence:** 90%
**Complexity:** Low–Medium
**Status:** Unexplored
