Solutions × Environments Grid
✓ Shipped environment-solution-matrix Conf 90% Cx Low
Shipped 2026-07-04. Implemented as a pivot of the existing flowline status command, not a new dashboard command — see docs/plans/2026-07-04-001-feat-status-dashboard-grid-plan.md. Landed via StatusGrid.BuildGridRows / StatusGrid.RenderGrid (src/Flowline/Commands/StatusCommand.cs, extracted into StatusGrid.cs), with follow-on additions beyond this idea's original scope: a repo column, drift detection, dirty indicator, and an icon legend.
Description

Pivot StatusCommand's rendering from nested (environment header → solutions indented beneath) to a Spectre.Console Table: solutions as rows, environments (Dev / Test / UAT / Prod) as columns, version strings as cells. This is a pure rendering change — the underlying data structure (Dictionary<string, string?> Versions per env) is already computed and discarded after the nested render. The grid makes cross-environment version comparison instant instead of requiring the reader to mentally track indented values across four separate env blocks.

This is the foundational idea — everything else in this list (health badges, anomaly detection, promote action) composes on top of the grid as additional cell annotations or row states. Building the grid first is the natural prerequisite.

Basis

direct StatusCommand.cs:65 — data computed as Dictionary<string, string?> Versions per env, available before rendering begins. StatusCommand.cs:105–127 — current nested render confirmed; no Table anywhere in codebase.

Rationale

The matrix is already computed — the work is in the renderer. Switching to a Spectre.Console Table gives the reader 4-environment comparison at a glance instead of mentally correlating 4 nested env blocks. This is the same pivot Azure DevOps and Power Platform Pipeline UI make when they show "environments as columns." The data model needs no change; the rendering pattern (AddColumn, AddRow per solution) is directly supported by Spectre.Console.

Downsides
Wide solution names truncate in narrow terminals. Spectre.Console Table requires all data before it can render (no streaming rows); acceptable since all 4 env queries run in parallel anyway.

Local-as-First-Column
✓ Shipped environment-solution-matrix Conf 90% Cx Low
Shipped 2026-07-04. Local column implemented alongside idea 1 in the same pass — see docs/plans/2026-07-04-001-feat-status-dashboard-grid-plan.md (R2, KTD4). Reuses DeployCommand.ReadLocalSolutionVersion; a solution with no local checkout yet renders a dash.
Description

Add a Local column as the leftmost column of the matrix, showing the version declared in the local Package/src/Other/Solution.xml. This read is purely offline — no network, no PAC CLI — and is available instantly even when environment queries are still in flight. The full matrix becomes: Local → Dev → Test → UAT → Prod.

When Local is ahead of Dev, the dashboard immediately shows an uncommitted version bump waiting to be deployed. This catches the common scenario where a developer has incremented the solution version locally but hasn't deployed yet — something the existing status output cannot show.

Basis

direct DeployCommand.cs:342 — ReadLocalSolutionVersion() already parses Package/src/Other/Solution.xml in production code. The offline parse capability exists; this idea surfaces it in the matrix.

direct CLAUDE.md folder-structure spec confirms Package/src/ holds the unpacked solution XML in all Flowline projects.

Rationale

The Local column extends the dashboard into "what's about to be deployed" territory — making the developer's working copy part of the pipeline view. It fills in the gap between a finished version bump and its first deployment, which currently requires the developer to check Solution.xml manually. Because ReadLocalSolutionVersion() is already written and tested, the implementation cost is minimal.

Downsides
Only meaningful when run from inside a Flowline project with a Package/src/ tree. Column shows nothing useful (dash) in global context or from CI. Needs a graceful absent-column path.

Import Health Badge
data-model-and-freshness Conf 75% Cx Medium
Description

Annotate each cell in the matrix with the outcome of the most recent import for that solution in that environment, queried from the Dataverse virtual table msdyn_solutionhistories. Each cell would show the version plus a compact badge: ✓ 2h ago (green), ✗ failed 45m ago (red), or — no history (gray). A failed badge surfaces the common silent-failure case: a version string that looks correct but whose import had errors visible only in a browser session.

The query requires a new SDK path (via IOrganizationServiceAsync2, following the SolutionReader.cs pattern) since StatusCommand currently uses PAC CLI only. The query filters msdyn_operation = 0 (Import), orders by msdyn_starttime descending, takes 1 row per solution per environment.

Basis

direct SolutionReader.cs:10–39 — established IOrganizationServiceAsync2 QueryExpression pattern to follow for the new query.

direct Zero matches for msdyn_solutionhistor in the codebase — entirely untapped data source.

direct Dataverse fields confirmed: msdyn_result (bool), msdyn_operation (0=Import), msdyn_starttime (datetime), msdyn_solutionname (for join).

Rationale

Version strings alone are a lagging indicator. An import can partially fail — leaving the version registered as updated but the solution in a broken state — and the only current way to know is opening the Power Platform admin center. The health badge makes the most recent import outcome a first-class cell annotation, surfacing failures that currently require a browser tab. This is the highest-novelty idea in the set: it unlocks a data source that Flowline does not touch today and that no other Flowline command would need without the dashboard.

Downsides
Requires a new Dataverse SDK authentication path per environment (can't use PAC CLI for virtual table queries). Adds latency. msdyn_solutionhistories is a virtual table — availability depends on environment licensing; needs graceful absent-table fallback.

--json as Versioned Schema
integration-and-extensibility Conf 90% Cx Low
Description

Add a --json flag to flowline dashboard that emits a stable, machine-readable document instead of an interactive table. The document carries a top-level schemaVersion: 1 field alongside the solution matrix (version per env), promotion-readiness per transition (once grounded in actual version comparison), and last-import status (if B2 is built). This is Flowline's first structured output — establishing the CI integration contract explicitly from day one.

In CI pipelines, flowline dashboard --json | jq '.solutions[] | select(.versionSkew)' becomes a lightweight deployment gate. The schemaVersion field lets consumers pin to a version, enabling Flowline to evolve the shape without silently breaking downstream scripts.

Basis

direct No --json or output-format flag exists on any Flowline command (confirmed).

direct ValidationCacheStore.cs:6–8 — System.Text.Json.JsonSerializer already imported and used; no new dependency.

direct StatusCommand.cs:24 — ConsoleHelper.IsInteractive() guard pattern established; --json routes the non-interactive path.

reasoned Explicit schemaVersion is the minimum contract discipline for a CI-consumed output — without it, any field rename breaks consumers silently.

Rationale

The matrix data the dashboard computes is exactly what a CI pipeline needs to gate a promote step or generate a deployment report. Exposing it as JSON with an explicit version contract makes the dashboard composable with existing tooling without requiring a full Flowline SDK. It is the lowest-effort integration path: same data, rendered differently, with one field (schemaVersion) committing to stability.

Downsides
Schema must remain stable once published — any field rename is a breaking change. The team needs to agree on the schema shape before shipping schemaVersion: 1, since evolving it post-ship requires bumping to schemaVersion: 2 and maintaining backward compatibility.

Exception-First Collapse
interaction-and-actions Conf 85% Cx Low
Description

Invert the default view assumption: rather than always showing all solutions, show only solutions with anomalies by default — version skew, failed last import, DTAP violation, not-yet-deployed in a target env. Solutions where everything is nominal collapse to a single dim footer line: "7 solutions nominal."

Keyboard shortcuts toggle the view: a expands all solutions, f shows only failures (a subset of anomalies). When running in --watch mode, newly anomalous solutions surface automatically. In CI or --json mode, the filter is irrelevant — output is always complete.

Basis

direct StatusCommand.cs renders all solutions with equal visual weight — no anomaly prioritization exists.

direct DtapGateOutcome enum at DeployCommand.cs:289 provides gate violation signals.

reasoned AnsiConsole.Live() + Console.ReadKey() toggle between two pre-computed renderables (expanded vs collapsed) — ReadKey is not in Spectre's prohibited-with-Live list (confirmed by verifier).

Rationale

As a project grows in solutions, the "show everything" default becomes noise. An ops board's primary job is to surface exceptions — the same principle behind logistics exception dashboards and production monitoring tools that only page on deviation. Exception-first makes the dashboard actionable: if the screen shows nothing, all is well. If rows are visible, something needs attention. The keyboard-expandable "show all" path preserves full visibility without making it the default cognitive load.

Downsides
Requires anomaly detection logic as a prerequisite (at minimum A4 version skew, or B2 import health). Empty screen on first use (all nominal) may confuse users who expect a table. Needs a clear explanation of what "nominal" means in the footer line.

Version Skew Anomaly Detection
deployment-pipeline-flow Conf 85% Cx Low
Description

After the version matrix is populated, compare version tuples across the DTAP chain per solution and flag inversions: a case where a higher environment (e.g., Prod) has a newer version than a lower one (e.g., Test). This indicates an out-of-band deployment that bypassed the normal DTAP promotion flow. The cell in the anomalous environment renders with a warning glyph and a distinct color to make the inversion immediately visible.

This is pure logic over already-fetched data: parse version strings with Version.Parse(), compare across the ordered Dev → Test → UAT → Prod sequence, flag any row where the ordering is violated. No new queries, no new SDK paths.

Basis

direct DeployCommand.cs:287–338 — ResolveDtapGate() only prevents forward promotion attempts; it does not detect existing inversions already in the environments. Confirmed by verifier: inversions bypass the gate entirely.

reasoned Version.Parse() handles Dataverse 4-tuple version strings (major.minor.build.revision). Comparison across the ordered env list (StatusCommand.cs:54–60) produces the inversion signal without additional infrastructure.

Rationale

An out-of-band deploy to Prod (e.g., an emergency hotfix pushed directly via the Power Platform admin UI) creates a version inversion that no existing Flowline command detects. Without detection, the dashboard would silently show a higher Prod version without flagging it as anomalous. Version skew detection makes the implicit "DTAP ordering violated" case explicit — the most important pipeline health signal that costs nothing to compute once the matrix is built.

Downsides
False positives possible: some teams intentionally deploy to selected environments out-of-order. The flag is informational, not a blocker, but teams should be able to suppress it per solution. Dataverse 4-tuple version comparison may behave unexpectedly with build-number-only increments (1.0.0.1 vs 1.0.0.0) vs minor increments.

Interactive Promote Action
interaction-and-actions Conf 70% Cx Medium
Description

After the matrix renders, pressing a key (e.g., p) surfaces a SelectionPrompt listing the viable promotion targets: solution + destination env combinations where version comparison and DTAP position indicate a promotion would succeed. Selecting an option confirms and runs the deploy inline, without leaving the dashboard context.

The key Spectre.Console constraint: AnsiConsole.Live() cannot run simultaneously with SelectionPrompt. The Live session must be explicitly stopped before the prompt is rendered, then the matrix re-renders after the promotion completes. The ConsoleHelper.IsInteractive() guard at StatusCommand.cs:24 skips the promotion UI in CI or piped contexts.

Basis

direct ProfileResolutionService.cs:37–54 — SelectionPrompt<T> usage pattern established in the codebase.

direct StatusCommand.cs:24 — ConsoleHelper.IsInteractive() guard; CI path is already distinguished.

reasoned Spectre.Console issue #1452: AnsiConsole.Live() and SelectionPrompt cannot run simultaneously. The Live session must be stopped, the prompt rendered, and a new Live session started after the deploy returns. This is a known constraint, not a speculative one.

Rationale

The dashboard answers "what's deployed where" — the natural next question is "can I promote X?" and the next action is "promote it." Collapsing the view→decide→act cycle into a single session removes the context switch to flowline deploy, which requires re-specifying solution names and target environments the dashboard already has in scope. This is the "click the Promote button in ADO Pipelines" analog for the CLI.

Downsides
The Live-stop/restart pattern adds UX jank (flash between states). Viable promotions require meaningful version comparison logic; if that logic is wrong, the SelectionPrompt offers false positives. Spectre #1452 is a real constraint — any regression in Spectre's threading behavior could break the handoff.

Rejection Summary

Deliberate Axis Gap — deployment-pipeline-flow

True per-solution pipeline-flow visualization (DTAP lane arrows, per-solution READY/BLOCKED) was attempted by C1, C2, and C4, but all rely on ResolveDtapGate() as their basis. The verifier confirmed a critical false premise: the gate is keyed on targetUrl alone — same env, same outcome for every solution. Per-solution lane coloring requires a direct version-comparison step (e.g., devVersion < testVersion?) not cited in any candidate's basis. A4 (version skew detection) partially covers this axis via anomaly detection on the matrix. A future idea for this axis would ground per-solution promotion readiness in version comparison, not the gate.
