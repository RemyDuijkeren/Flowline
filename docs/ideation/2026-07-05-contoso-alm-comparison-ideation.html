What Microsoft's Contoso Real Estate ALM Reference Earns Flowline

Microsoft's contoso-real-estate-power-platform (Scott Durow's reference architecture for developing, deploying, and managing a Dataverse solution via Power Platform, Azure, and GitHub) is the enterprise-managed-ALM answer to the same problem Flowline solves fast and unmanaged. This pass reads the live contoso repo line-by-line against Flowline's actual source to find what's worth adapting — and names what should stay out on purpose.

Codebase Context

Flowline CLI (current state)

C# .NET 10 CLI, v0.9.0 (released 2026-06-29, v1.0 targeted 2026-07-20). Commands: clone/push/sync/deploy/provision/generate/status. Convention-over-configuration (attribute-driven plugin registration), PAC-CLI-auth-reuse, AI-native (DATAVERSE_CONTEXT.md).

Already shipped, verified by reading source: non-bypassable DTAP gate in DeployCommand (tier promotion, System.Version comparison); a managed/unmanaged type guard; an IPostDeployService DI fan-out with three implementers — SolutionCheckService (already runs pac solution check pre-import!), BackupService, and a two-phase OrphanCleanupService that already computes cross-solution component membership. Fully automated semver via MinVer + git tags (no manual bump needed).

Confirmed gaps: ProjectSolution has only Name/IncludeManaged/Generate — no dependency/order field; ProjectConfig.Solutions is an unordered HashSet; PushCommand.cs has zero references to the Solution Checker; CI auth uses stored service-principal secrets, no OIDC.

contoso-real-estate-power-platform (verified via GitHub API/raw fetch)

Three segmented solutions (Controls → Core → Portal) driven by a Dataverse dependency-versioning constraint (PCF components must be managed and pre-built before a canvas app can reference them) — not a general multi-solution design pattern.

CI/CD: GitHub Actions, federated identity (pac auth create --githubFederated), per-solution semantic versioning via changeScope-scoped commit tags, pac solution check parsing SARIF with a Critical/High threshold, GitHub Environments with required-reviewer approval gates, deploymentSettings.json synthesized at deploy time from a GitHub Actions variable (never committed), azd/Bicep for the companion Azure Functions payments API.

Live-fetched fact that reshapes the multi-solution question: SOLUTIONS_CONFIG is a flat, manually ordered JSON array; the deploy matrix runs it with max-parallel: 1 — no dependency graph, no topological sort exists anywhere in contoso's own repo.

Market signal (external research, value: high)

2026 community consensus (Microsoft Learn, engineering blogs) treats full custom DevOps construction as unnecessary overhead for most teams — minimal ALM (environment separation, version control, approval-gated deploy, rollback) is the mainstream position, not Flowline's fringe stance. ALM Accelerator for Power Platform is now deprecated by Microsoft; spkl is cited as prior art in Flowline's exact "simple ALM" niche. Managed Environments (Feb 2026+) auto-enforce Solution Checker on import regardless of what CI already ran — a platform-level constraint no tool can opt out of.

1

Multi-solution deploy ordering — pick one of three designs

Top pick multi-solution-topology Confidence: High Complexity: Medium
Description
Flowline's ProjectSolution has only Name/IncludeManaged/Generate — confirmed by direct read, no dependency or order field exists — and ProjectConfig.Solutions is an unordered HashSet, confirmed to have no deterministic enumeration order at all. That's a real correctness gap the moment a project has two solutions with an import dependency between them. Three independently-proposed designs solve it, and they're genuinely in tension:

(A) Declared dependency graph — add Dependencies: string[] to ProjectSolution; deploy/push topologically sort before acting.
(B) Flat declared order + change-scoped skip — an ordered array in .flowline, plus a --all flag that iterates it sequentially and skips solutions whose folder didn't change.
(C) Error-driven emergent ordering — zero config surface at all; deploy --all attempts every solution, catches Dataverse's own missing-dependency import error and retries failures in a second pass — reusing the exact catch-and-defer pattern already proven in OrphanCleanupService.
Basis
direct: live-fetched contoso build.yml/deploy-environment.yml confirm SOLUTIONS_CONFIG is a flat GitHub Actions variable JSON array consumed by a matrix with max-parallel: 1 — no topological sort or dependency field exists anywhere in contoso's own workflow YAML, verified by direct fetch, not assumption. direct: ProjectSolution.cs confirmed to hold only Name/IncludeManaged/Generate; ProjectConfig.cs confirmed Solutions is a HashSet with no order guarantee. direct: OrphanCleanupService's two-phase catch-on-error-code-0x80047002-and-defer pattern is confirmed, proven architecture (docs/solutions, high severity) that option (C) reuses directly.
Rationale
This is the single most consequential design fork in the whole comparison: even Microsoft's own flagship multi-solution example doesn't build a dependency graph — it gets away with a flat ordered list. That's hard evidence against reaching for graph/topological-sort machinery by default (option A), and evidence for either the equally lightweight flat-order approach (B) or the zero-config-surface approach (C) that best matches Flowline's convention-over-configuration philosophy. This isn't a "nice to have" — it's the one Flowline currently has zero answer for.
Downsides
Option (A) is the most powerful but adds a config surface Flowline has consistently avoided elsewhere, and contoso's own repo doesn't validate the extra complexity. Option (C) needs careful scoping of retry semantics (how many passes, what counts as a permanent vs. transient failure) before it's safe to ship. Whichever option ships, do not let it grow into "spin up an environment per solution" — that overhead exists in contoso only to satisfy a managed-solution boundary constraint Flowline doesn't share.

2

Shift Solution Checker left to push, not just deploy

solution-checker-gate Confidence: High Complexity: Low
Description
SolutionCheckService is wired exclusively into DeployCommand's IPostDeployService fan-out. A grep of the full PushCommand.cs for SolutionCheckService/IPostDeployService returns zero matches — confirmed by direct read. That means the fastest point in the dev loop never sees checker feedback; a solo developer only learns about a Critical finding at deploy time, potentially after several pushes. Add an optional fast local check at push time against the unpacked solution folder, before packaging.
Basis
direct: full read of PushCommand.cs (415 lines) confirms zero references to SolutionCheckService or IPostDeployService — verified independently by two ideation frames and confirmed exactly by the Phase 3 verifier.
Rationale
Directly serves the "zero CI minutes, solo dev, no team to approve" end of Flowline's positioning — the check becomes a local dev-loop habit instead of a CI-stage gate, the opposite of contoso's pipeline-stage model. The hard part (running pac solution check, interpreting the result) is already built; this is a wiring change, not new logic.
Downsides
Running the checker on every push adds real latency (~30-90s) to the fastest command in the loop — should almost certainly be opt-in (a flag or config toggle), not default-on.

Cross-solution impact preview (flowline status --cross-solution)

multi-solution-topology Confidence: High Complexity: Low
Description
OrphanCleanupService already computes cross-solution component membership (which components a solution shares with sibling solutions) to decide RemoveFromSolution vs Delete during cleanup — but only reactively, after a deploy has already happened. Expose that same computation as a pre-deploy diagnostic: "if I deploy solution B, what does it share with solution A, and what would this touch."
Basis
direct: OrphanCleanupService.GetCrossSolutionMembershipAsync confirmed by direct read to genuinely compute solution-component cross-membership, currently consumed only inside RunPreImportAsync/RunPostImportAsync.
Rationale
Reuses computation Flowline already built and pays for, just repointed to run earlier and surface to the user instead of being thrown away until cleanup time. Near-zero net-new logic for a real diagnostic gap.
Downsides
Only useful once a project actually has 2+ solutions with genuine overlap — low priority for the common single-solution case, but cheap enough that this isn't really a cost.

4

Per-tier Solution Checker severity threshold

solution-checker-gate Confidence: High Complexity: Low
Description
Today the checker gate is binary and global: SolutionCheckService.ShouldAbort only ever aborts on criticalCount > 0, for every tier. But DeployCommand's DTAP gate already resolves a full tier-ordering concept (Dev < Test < UAT < Prod) for version gating. Reuse that same tier hierarchy to vary checker strictness by target: Dev/Test tolerate Medium findings, UAT blocks High+, Prod blocks Critical+High — one config value per tier, not a new subsystem.
Basis
direct: SolutionCheckService.cs confirmed hardcoded threshold ShouldAbort(int criticalCount) => criticalCount > 0; DeployCommand's tier/predecessor resolution logic confirmed already implemented for version gating (docs/solutions, DTAP gate, high severity). Independently proposed by two separate ideation frames working from the same evidence — cross-validated.
Rationale
The DTAP tier hierarchy is architecture already paid for once; extending it to severity gating is near-zero net-new plumbing compared to contoso's separate environment-tier config layer, and answers the real complaint that Dev and Prod get identically lenient checker gates today.
Downsides
Touches the .flowline config schema and PostDeployContext shape (which currently carries no tier field) — small scoping conversation needed before building.

5

Cache Solution Checker results by content hash

solution-checker-gate Confidence: Medium Complexity: Medium
Description
Neither Flowline nor contoso caches pac solution check output — contoso's build.yml re-runs it fully every build (confirmed: no hash/skip logic in the fetched workflow step). Key a cache on (packed-zip content hash, checker ruleset version, target tier) so a repeat deploy of an unchanged package skips a 30-90s re-run. While touching this service, also wire the missing exclude-files plumbing: PacUtils.CheckSolutionAsync has no -ef/exclude parameter at all today, so Flowline can't yet do what contoso does manually (-ef bundle.js to suppress transpiled-JS false positives) — auto-derive the exclude list from the existing WebResources/dist/ gitignored-build-output convention instead of asking users to hand-maintain one.
Basis
direct: PacUtils.CheckSolutionAsync confirmed to have no caching and no exclude-files parameter wired today. external: contoso's build.yml `pac solution check -ef bundle.js` — hand-maintained exclude list, no caching. reasoned: the packed zip is already deterministic pre-check, making content-hash keying straightforward; must key on ruleset version too since a cache can only ever skip Flowline's own gate, never the platform's Managed-Environment-enforced one.
Rationale
Cuts real deploy latency for the common "nothing changed in this solution, just deploying its sibling" case on multi-solution repos, and closes a plumbing gap (no exclude-list support exists at all yet) while the service is already being touched.
Downsides
Cache-invalidation correctness (ruleset-version bumps, partial-content changes) needs real design scrutiny before shipping — this is exactly the kind of tradeoff that benefits from a second pair of eyes.

6

Build-once, promote-many: immutable artifact across DTAP tiers

ci-auth-versioning Confidence: High Complexity: Medium
Description
Contoso versions each solution independently, packs it once, and publishes it as a tagged GitHub Release asset — the same zip is promoted dev→test→prod, never rebuilt. Flowline's DeployCommand calls PackSolutionAsync fresh from local source XML on every deploy invocation, so two deploys of "the same version" to different tiers aren't guaranteed to be identical bytes. Add a --from-artifact <zip> flag that skips re-packing and lets the existing DTAP gate verify the artifact's embedded version instead.
Basis
direct: contoso build.yml (softprops/action-gh-release, tag-named asset) + import-solution.yml (regex-matched tag download) confirmed via live fetch. direct: DeployCommand.cs confirmed PackSolutionAsync runs unconditionally, no artifact-reuse path exists today. external: build-once/promote-immutable-artifact is standard practice (OCI images, npm publish).
Rationale
The DTAP gate checks version number only — it never confirms the artifact hitting Prod is bit-identical to what passed UAT. This closes an integrity gap contoso solves with heavier machinery (GitHub Releases); Flowline can solve it with one local flag instead.
Downsides
Distinct from the already-proposed Git-Tag-as-approval-gate idea (that's about permission to promote; this is about artifact identity) — worth keeping the two ideas separate in any future design doc so they don't get conflated.

7

OIDC-native CI auth via the existing token-provider callback

ci-auth-versioning Confidence: Medium Complexity: Medium
Description
DataverseConnector confirmed by direct read to hand ServiceClient a token via an async _ => token callback constructor — a mechanism generic enough to accept any bearer token, not only one sourced from PAC CLI's own MSAL cache. Today CI auth goes through pac auth create --kind ServicePrincipal --clientSecret (PAC's own profile store, read via ConnectServicePrincipalAsync), which requires a stored client secret. Add a CI-only connect path that acquires an Entra token externally (via azure/login + a federated-credential app registration) and feeds it directly into the existing callback — bypassing PAC's profile lookup for that path entirely, without waiting on PAC CLI itself to ever add OIDC support.
Basis
direct: DataverseConnector.cs token-provider callback confirmed by the Phase 3 verifier via full read. Correction from verification: the mechanism is not the SecretResolver.cs chain (that class only serves XrmContext code generation, unrelated to deploy auth) — the actual current SP-auth path is PAC's own stored profile via ConnectServicePrincipalAsync. external: OIDC/federated identity is MS's own documented standard for GitHub Actions → Power Platform (microsoft/powerplatform-actions v1.5.0 + azure/login), but pac auth create itself has no --federatedToken flag today (only --managedIdentity).
Rationale
Two ideation frames drew different conclusions from adjacent facts here, worth naming explicitly: one frame concluded OIDC is not buildable until PAC CLI itself ships a federated-credential flag; another found that Flowline's own auth-consumption code doesn't actually require going through PAC's stored profiles at all — the token-provider callback is auth-mechanism-agnostic. Both facts are true simultaneously; the second reading is the sharper one, because it identifies a path that doesn't wait on an upstream PAC CLI feature that doesn't exist yet.
Downsides
This needs a short design spike to confirm the callback route works end-to-end with an externally-acquired token before committing — the confidence is Medium, not High, specifically because that spike hasn't happened yet. This would be a new CI-only code path alongside PAC-based auth, not a pac auth create feature.

Generated relationship diagram in DATAVERSE_CONTEXT.md

architecture-docs Confidence: High Complexity: Low
Description
DataverseContextGenerator.ReadRelationships already parses EntityRelationship XML per entity and emits a markdown relationships table into DATAVERSE_CONTEXT.md, regenerated fresh on every sync/clone. Add a second renderer over the same already-parsed data: a Mermaid entity-relationship diagram, generated — never hand-drawn like contoso's static docs/06-design.md C4 diagrams, which drift the moment the schema changes since nothing regenerates them.
Basis
direct: ReadRelationships confirmed by direct read (parses OneToMany/ManyToOne/ManyToMany, already builds a table); GenerateAsync confirmed called from both SyncCommand.cs and CloneCommand.cs. external: contoso's docs/06-design.md ships hand-authored, git-committed Mermaid C4 diagrams that never regenerate.
Rationale
Independently proposed by three separate ideation frames working from the same evidence — the strongest convergence signal in this entire set. Zero new extraction code: a second output over data already parsed, not a new subsystem. Directly serves Flowline's AI-native pitch — a flat 625-line markdown list partially defeats the "agents shouldn't have to reconstruct structure by reading" goal for relationship-heavy solutions.
Downsides
Adds Mermaid noise to every DATAVERSE_CONTEXT.md diff on schema change — consider a flag on the existing generation step rather than always-on, for teams that don't want it.

9

Managed Environment awareness warning

solution-checker-gate Confidence: Medium Complexity: Low
Description
Managed Environments (Feb 2026+) will auto-enforce Solution Checker on every import at the platform level, regardless of what Flowline's own SolutionCheckService already ran. A user who bypasses or soft-fails Flowline's existing gate today can get blindsided later when an org admin flips the environment to Managed — same solution, same code, a new platform-level block with zero warning from Flowline. Have status/deploy query org settings (via the existing DataverseConnector) for the Managed Environment flag and surface a warning if the local gate was bypassed against it.
Basis
direct: IPostDeployService fan-out and its three implementers confirmed by direct read. external: Managed Environments' Feb 2026 forced-checker enforcement, per external research grounding.
Rationale
Operator education, not a new blocking control — respects "don't rebuild what the platform already enforces." The explicit companion non-goal matters as much as the warning itself: do not build detection/workaround code that tries to replicate or bypass the platform's own enforcement — one paragraph in Known-Limitations.md plus this warning is the right scope, nothing more.
Downsides
Requires an org-settings read Flowline doesn't do today — small, but it's a new capability, not free.

10

flowline status --drift <env>

config-data-import Confidence: Medium Complexity: Medium
Description
Contoso's PAC_DEPLOY_CONFIG is a GitHub Actions variable hand-edited through the Settings UI — opaque, not diffable, no PR review on config changes. The prior Flowline ideation doc already proposed the better alternative: a git-tracked flowline.env.<name>.json. This adds a genuinely new capability on top of whichever settings-file design lands: a non-blocking command that diffs the committed file against the live environment's actual connection-reference/env-variable values (reusing service calls sync already makes).
Basis
direct: contoso's variable-storage mechanism (quoted from docs/04-ci-cd-setup.md, live-fetched) has no diff/review path. external: Terraform plan/drift detection — the entire IaC discipline exists to replace exactly this "hand-edited external config" anti-pattern.
Rationale
Config drift (someone hand-edits a connection reference live in the maker portal) is invisible today in both tools; contoso doubles the exposure by keeping its source of truth outside git entirely. This is the one genuinely new angle on an axis two prior ideation docs already covered in depth (16 ideas) — everything else on this axis was a restatement and was cut (see Rejection Summary).
Downsides
Depends on the eventual settings-file design from the prior ideation doc actually shipping first — this isn't standalone. "Advisory or blocking" is a real product call to make before building.

11

CI-native exit-code rendering

ci-auth-versioning Confidence: High Complexity: Low
Description
ExitCode.cs states its codes are "a stable public API — agents and scripts pattern-match on these values," and each value carries remediation text as an XML doc comment (e.g. PartialSuccess: "Check output for items to remove manually via maker portal"). None of that reaches a CI run as anything other than buried console text. Detect $GITHUB_ACTIONS/$TF_BUILD and render the existing remediation strings as ::error:: annotations or a step summary.
Basis
direct: ExitCode.cs confirmed to carry exact remediation text as doc comments on every value, unused outside console output today.
Rationale
The hard part — deciding what each failure means and how to fix it — is already written. Only rendering is missing. Small, cheap, no new copywriting, no new subsystem — genuinely a quick win rather than a design decision, which is why it ranks last despite scoring High confidence.
Downsides
Loosest axis fit of the set — this is CI output ergonomics more than auth or versioning specifically; kept on this axis since it's CI-facing.
