- **P2 (maintainability, confidence 75)** — `src/Flowline.Core/Services/FormEventRenameAdvisor.cs:19`: `FormEventRenameAdvisor.Suggest` and `FormEventReader.BuildFormNotFoundMessage` take a raw 5-field positional tuple `(Guid Id, string Name, string EntityLogicalName, string FormXml, string? A positional tuple is an easy place for a future edit to transpose `Name`/`EntityLogicalName`. Suggested fix: retype the new parameters to `IReadOnlyList<DataverseForm>`.

- **P2 (adversarial, confidence 75)** — `src/Flowline.Core/Services/FormEventRenameAdvisor.cs:52-88` (`FindSelfTagMatch`): returns on the *first* sharing annotation that produces a self-tag match, without checking whether another sharing annotation (e.g. `onSave` vs `onLoad` on the same form pair) has its own genuine self-tag evidence pointing at a *different* candidate. `BuildSuggestion` then prints a rewrite line for every sharing annotation using that single winning candidate — so in a rare disagreement case, the tool could confidently tell a developer to repoint an annotation whose own evidence was never checked. This is a behavior change to the matching algorithm (favor recall vs. precision), not a mechanical fix, so it needs a design call rather than auto-apply.

## Accepted, not a residual (documented tradeoff, no action needed)

- **P2 (adversarial, confidence 100)** — `FormEventIdentityCache.SetMany` does an unlocked read-modify-write; two concurrent `flowline push` runs against the same environment can clobber each other's cache writes. This is an already-accepted design tradeoff from the plan (`docs/plans/2026-07-12-001-feat-form-event-rename-resilience-plan.md`, KTD2): the cache is advisory bookkeeping, a lost write just means a future push re-resolves by name, and corrupt JSON already degrades to a clean cache-miss rather than crashing. Adding file locking would be disproportionate for this feature.
