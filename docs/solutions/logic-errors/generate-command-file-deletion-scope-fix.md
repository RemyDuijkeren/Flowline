# generate command deletes user-owned files during temp-swap

## Problem

`flowline generate` uses a temp-swap pattern (generate into `Models~`, delete `Models`, rename `Models~` → `Models`) that treated the entire output folder as generator-owned, silently destroying any user-placed files (.csproj, hand-written partials, README) alongside the generated ones.

## Symptoms

- User-created files in `Models/` vanish after `flowline generate` — no error, no warning
- Pointing `--output` at the project root on first run permanently deletes the `.csproj`
- The uncommitted-changes warning said "they will be overwritten" regardless of file type
- No recovery path: once `Directory.Delete(modelsFolder, recursive: true)` runs, files are gone

## What Didn't Work

- **Manifest/sidecar approach** — tracking generated files in a separate state file: rejected, adds persistent state to maintain and synchronize; stateless marker detection handles stale-entity cleanup automatically without any new state
- **`overwrite: true` on `File.Copy`** — wrong semantics: would overwrite the freshly-generated file with the user's older copy at the same relative path; generator output must win on collision
- **Markdown marker detection** — `DATAVERSE_CONTEXT.md` uses a Flowline marker but lives outside the `Models` swap path; out of scope and a separate concern
- **`if (!File.Exists(dest)) File.Copy(file, dest)` collision guard** — first fix attempt after the IOException crash on `XrmExtensions.cs`. Suppresses the crash (generator version already at `dest`, skip fires) but for the wrong reason — it's a symptom treatment, not an ownership fix. A user file that legitimately shares a path with future generator output would be silently discarded with no indication. Removed in favor of `generatorPaths.Contains(rel)` as the primary ownership signal.
- **Header-only detection via `IsGeneratorOwned` alone** — insufficient: `XrmExtensions.cs` from XrmContext3 is generator-produced but carries no `<auto-generated>` header. `IsGeneratorOwned` returns `false`, treating it as user-owned and causing an IOException when copied over the freshly-generated version already in `tempFolder`. The `generatorPaths` set from `tempFolder` resolves this without any generator-specific special casing.

## Solution

Pre-copy-then-swap: before deleting `Models`, copy every non-generator-owned file into `Models~` at its relative path, then proceed with the existing delete + rename unchanged.

Two ownership signals determine whether a file is generator-owned. Both are checked; a file is copied only if both return false:

1. **`generatorPaths`** (primary) — a `HashSet<string>` of all relative paths the generator wrote to `Models~`. Any file at a matching path is definitively generator-owned, regardless of content. This handles generator files without the standard header (e.g. `XrmExtensions.cs` from XrmContext3).
2. **`IsGeneratorOwned`** (secondary, stale detection) — reads the first 15 lines for `<auto-generated>`. Files carrying the marker that are absent from `generatorPaths` are stale (entity removed from Dataverse) and are deleted with the old folder.

```csharp
// GenerateCommand.cs — shared tail (all generators)
var generatorPaths = Directory.EnumerateFiles(tempFolder, "*", SearchOption.AllDirectories)
    .Select(f => Path.GetRelativePath(tempFolder, f))
    .ToHashSet(StringComparer.OrdinalIgnoreCase);

if (generatorPaths.Count == 0)
    throw new FlowlineException(ExitCode.BuildFailed,
        "Generator reported success but produced no output. Re-run with --verbose to see tool output.");

if (Directory.Exists(modelsFolder))
{
    foreach (var file in Directory.EnumerateFiles(modelsFolder, "*", SearchOption.AllDirectories))
    {
        var rel = Path.GetRelativePath(modelsFolder, file);
        if (generatorPaths.Contains(rel) || IsGeneratorOwned(file)) continue;
        var dest = Path.Combine(tempFolder, rel);
        Directory.CreateDirectory(Path.GetDirectoryName(dest)!);
        File.Copy(file, dest);
    }
    Directory.Delete(modelsFolder, recursive: true);
}
Directory.Move(tempFolder, modelsFolder);
```

`IsGeneratorOwned` reads lines without a try/catch — `File.ReadLines` on binary files returns garbled chars that never match `<auto-generated>`, so no encoding guard is needed:

```csharp
private static bool IsGeneratorOwned(string filePath) =>
    File.ReadLines(filePath).Take(15).Any(line => line.Contains("<auto-generated>"));
```

Also fixed the uncommitted-changes warning to be accurate (`GenerateCommand.cs:187`):

```
Before: "has uncommitted changes — they will be overwritten. Commit them first if you want to review what changed."
After:  "has uncommitted changes — Flowline replaces generated files, preserves user files. Commit first to review the diff."
```

**Code review catches:**
- Initial `IsGeneratorOwned` used `catch { return false; }` — removed. Binary files never throw on `File.ReadLines`; real I/O errors (`IOException`, `UnauthorizedAccessException`) now bubble to the global handler as intended.
- Initial merge step called `Directory.EnumerateFiles(tempFolder)` twice — once for `.Any()` empty-output check, again to build `generatorPaths`. Combined into a single scan: build `generatorPaths` first, check `Count == 0` instead.

## Why This Works

Root cause: the swap pattern assumed exclusive generator ownership of the output folder. The fix adds a pre-pass with two ownership signals:

1. **Generator output set** (`generatorPaths`) — built from `tempFolder` after the generator runs. Any file path the generator produced is definitively its own, regardless of content. This resolves `XrmExtensions.cs`: generator-produced but header-free, so header-only detection failed; path-set detection is unambiguous.

2. **`<auto-generated>` header** — catches stale generated files: files that carry the marker but are absent from the current run's output (entity removed from Dataverse). These are correctly deleted with the old folder without needing any explicit stale-tracking.

Files where both signals are absent are user-owned and rescued into `Models~` before the destructive delete. If the pre-copy step throws, `modelsFolder` is still intact — error propagates cleanly with no partial-state corruption. An outer `try/catch` around the entire post-generator section (build `generatorPaths` → merge → delete → move) deletes `tempFolder` on any failure before re-throwing, ensuring no temp folder is left behind.

## Prevention

- `GenerateFileSafetyTests` class (`tests/Flowline.Tests/GenerateCommandTests.cs`) explicitly covers:
  - User `.csproj` in `Models/` survives generate run
  - Hand-written partial class survives generate run
  - Stale generated file (no new output) is deleted
  - Generator output wins at a collision path — no `IOException` thrown
  - Binary file in `Models/` treated as user-owned, survives
  - `IsGeneratorOwned` returns `true` for PAC and XrmContext headers
  - `IsGeneratorOwned` returns `false` for plain `.cs`, `.csproj`, `.md`, binary files
  - First-run case (no existing `Models/`) completes without error
- Any future directory-swap pattern that outputs into a folder the user may also write to should apply the same pre-copy rescue pass before `Directory.Delete`
- Use `generatorPaths.Contains(rel)` (built from the actual generator output) as the primary ownership signal — never rely solely on file content markers, as not all generator-produced files carry a header (`XrmExtensions.cs` is the canonical example)
