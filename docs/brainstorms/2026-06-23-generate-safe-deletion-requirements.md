# Generate Safe Deletion Requirements

**Date:** 2026-06-23
**Status:** Ready for planning

## Problem

`flowline generate` uses a temp-swap pattern: generates into `Models~`, deletes `Models` entirely, renames `Models~` to `Models` (`GenerateCommand.cs:295–297`). Any file the user placed in `Models` — a hand-written partial class, a `README`, a `.csproj` — is silently destroyed. The uncommitted-changes warning exists but only fires when git already tracks the file; a brand-new file in the folder gets no warning and no mercy.

Pointing the output at the project root (the natural first attempt) loses the `.csproj` permanently.

## Goal

A code generator must only delete what it owns. Files Flowline did not generate must survive a generate run, including across stale-entity cleanup.

## Ownership Signal

The auto-generated header present in every PAC and XrmContext output file marks generator ownership. Specifically:

- **C# files** — PAC and XrmContext3 both emit the same standard block within the first ~10 lines:
  ```
  //------------------------------------------------------------------------------
  // <auto-generated>
  //     This code was generated by a tool.
  // </auto-generated>
  //------------------------------------------------------------------------------
  ```
  PAC may also prepend `#pragma warning disable CS1591` on line 1. Both generators use this pattern across entity files, option set files, and the context root (`XrmContext.cs`). Detection: file contains `<auto-generated>` within its first 15 lines.

A `.cs` file that does not carry the marker is user-owned. Flowline must not delete it.

## Requirements

1. **Owned files only** — Before or during output folder replacement, Flowline deletes only files that carry the auto-generated ownership marker. Files without the marker are preserved.

2. **Stale cleanup still works** — A generated file for an entity that no longer exists in the solution (stale) is deleted because it carries the marker and has no corresponding new output. This is unchanged behavior; only unowned files get the new exemption.

3. **All generators in scope** — The rule applies to PAC, XrmContext, and XrmContext3 generators. Each already emits the standard C# `<auto-generated>` header; no generator-level change is required.

4. **Non-text files treated as user-owned** — Binary files cannot carry a text header. Treat any non-text file in the output folder as user-owned and leave it untouched.

5. **Atomic on the generated set** — If the generator fails mid-run, the user's pre-existing unowned files must remain intact. The failure recovery path must not delete unowned files.

6. **No new state file** — Ownership is determined by reading the file content at deletion time. No manifest, lock file, or sidecar is introduced.

## Scope

**In:** `GenerateCommand.cs` swap logic (lines 295–297 and the error recovery at 284–286). The temp-folder generation strategy itself is unchanged — only the merge/swap step changes.

**Out:**
- Generator output format (PAC/XrmContext/XrmContext3 output is not modified)
- Warning UX — the existing uncommitted-changes warning is separate and not changed here
- Non-`Models` folders — `.flowline`, `Package/src`, `WebResources` are unaffected
- `DATAVERSE_CONTEXT.md` — Flowline's own AI context file lives outside the Models folder and is not part of the Models swap; ownership detection for it is a separate concern

## Success Criteria

- A `.csproj` placed in the Models output folder survives `flowline generate`.
- A hand-written partial class in the output folder survives `flowline generate`.
- A generated file whose entity was removed from Dataverse is still deleted on the next `flowline generate`.
- No manifest or sidecar file is created.

## Assumptions

- PAC and XrmContext3 header format verified against real output. Both use `<auto-generated>` within the first 10 lines; PAC optionally prepends `#pragma warning disable CS1591`. Header detection against first 15 lines is sufficient and safe.
- The output folder may contain subdirectories (e.g., `Entities/`, `OptionSets/` in PAC output). Deletion logic must recurse. Non-`.cs` files in subdirectories without a text header are treated as user-owned.
